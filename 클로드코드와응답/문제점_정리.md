## 🔴 높음

### 1. 업로드 이미지가 재배포 시 소실됨 (운영 데이터 유실)
- **위치**: `businesscard_qr/src/main/resources/application.yml`(`file.upload-dir: ./uploads`),
  `card/service/BusinessCardService.java`(`storeImage`, `Files.copy(... targetPath)`)
- **내용**: 명함 이미지를 컨테이너 **로컬 디스크**(`./uploads`)에 저장. Railway 컨테이너는 파일시스템이
  **휘발성(ephemeral)** 이라 재배포·재시작·스케일아웃 시 저장된 이미지가 모두 사라짐.
- **영향**: 배포 후 시간이 지나면 기존 명함의 이미지 및 이미지 다운로드가 404. DB에는 경로만 남아 불일치.
- **관련**: 게이트웨이의 `application-prod.yml`에 `supabase.enabled: true`(외부 스토리지) 설정 흔적이
  있으나 **실제 코드에 연동은 없음** → 원래 외부 스토리지로 가려던 의도가 미완성 상태로 보임.
- **참고 방향**: S3/Supabase 같은 외부 오브젝트 스토리지 사용, 또는 Railway 볼륨 마운트.

### 2. 운영에서 H2 콘솔 기본 활성화
- **위치**: `businesscard_qr/src/main/resources/application.yml`
  ```yaml
  spring.h2.console.enabled: ${SPRING_H2_CONSOLE_ENABLED:true}   # 기본값 true
  ```
- **내용**: H2 콘솔은 임의 JDBC URL로 접속 가능한 범용 DB 콘솔. 기본값이 `true`라
  운영에서 `SPRING_H2_CONSOLE_ENABLED=false`를 명시하지 않으면 활성 상태.
  businesscard_qr 서비스가 자체 공개 URL을 가지므로(게이트웨이는 `/h2-console`을 프록시하지 않지만
  businesscard_qr 서비스 URL로 직접 접근은 가능) 노출 위험.
- **영향**: DB 콘솔/스키마 노출 가능성. (Spring Security frameOptions로 UI가 일부 막힐 수 있으나
  기본값을 운영에서 켜두는 것 자체가 위험.)
- **참고 방향**: 운영 프로파일에서 기본 비활성화.

## 🟡 중간

### 3. `/uploads/**` 가 인증·토큰 없이 공개되어 다운로드 토큰 설계가 무력화됨
- **위치**: `common/config/SecurityConfig.java`(`anyRequest().permitAll()`),
  `common/config/WebConfig.java`(`/uploads/**` 정적 리소스 핸들러)
- **내용**: 이미지 다운로드는 5분 만료 토큰(`/downloads/image?token=`)으로 보호하도록 설계돼 있지만,
  같은 이미지가 `/uploads/business-card-images/<파일명>` 경로로 **토큰·인증 없이 그대로** 서빙됨.
  결국 이미지 기밀성은 "토큰"이 아니라 "파일명 추측 난이도"에만 의존.
- **영향**: 토큰 기반 접근 통제의 실효성 저하. (파일명이 UUID+timestamp라 즉시 위험하진 않음)
- **참고 방향**: 두 접근 경로의 보호 수준을 일치시키거나, 정적 서빙을 막고 토큰 경로로만 제공.

### 4. 게이트웨이가 본문 전체를 메모리에 적재
- **위치**: `gateway/controller/ApiGatewayController.java`(`@RequestBody byte[] body`,
  응답 `byte[]`), `gateway/config/GatewayClientConfig.java`(`RestTemplate`)
- **내용**: 업로드(최대 50MB)와 다운로드(이미지/vCard) 본문을 통째로 `byte[]`로 버퍼링해서 중계.
- **영향**: 동시 요청/대용량 전송 시 힙 사용량 급증 → OOM 위험. 스트리밍 대비 메모리 비효율.
- **참고 방향**: 스트리밍 프록시(WebClient/reactor, 또는 Spring Cloud Gateway) 고려.

### 5. 카카오 API 호출에 타임아웃 없음
- **위치**: `auth/service/AuthService.java`(`HttpClient.newHttpClient()`, `httpClient.send(...)`)
- **내용**: 카카오 사용자 정보 조회에 connect/read 타임아웃을 설정하지 않음.
- **영향**: 카카오 응답 지연 시 로그인 요청 스레드가 오래 점유될 수 있음.
- **참고 방향**: `HttpClient.connectTimeout` + `HttpRequest.timeout` 설정.

### 6. 게이트웨이에 옛 애플리케이션 잔재 리소스가 대량 잔존
- **위치**: `dist_api_gateway/src/main/resources/`
  - `data-comment.sql`, `data-community.sql`, `data-files.sql`, `data-init.sql`,
    `data-rooms.sql`, `data-users.sql`
  - `templates/`(community, stomp, login, signup, mypage, oauth2-redirect 등 HTML)
  - `logback-spring.xml`
- **내용**: "게이트웨이 최소화" 때 Java 코드는 지웠으나 리소스는 남음. 특히 `logback-spring.xml`은
  **실제로 로딩되어** 게이트웨이가 `logs/application.log`·`logs/error.log`(휘발성 디스크)에 파일 로깅하고,
  존재하지도 않는 `org.hibernate.SQL`/`org.springframework.security`를 DEBUG로 설정(무의미한 노이즈).
  `data-*.sql`은 비표준 파일명이라 자동 실행되진 않지만 혼란 유발.
- **영향**: 혼란/오해, 불필요한 로그 파일 생성, 이미지 크기 증가.
- **참고 방향**: 게이트웨이에서 사용하지 않는 리소스 제거.

### 7. `application-prod.yml`이 게이트웨이에 있으나 실제로는 businesscard_qr용 + 미사용
- **위치**: `dist_api_gateway/src/main/resources/application-prod.yml`
- **내용**: 이 파일은 datasource(MariaDB), 카카오/구글 OAuth2 클라이언트, Supabase, JWT, 쿠키 설정 등
  **게이트웨이와 무관한** 내용을 담고 있음. 게이트웨이는 datasource·security 스타터도 없고 `prod`
  프로파일도 활성화되지 않아 **로딩되지 않는 죽은 설정**. 위치도 잘못됨(내용상 businesscard_qr 대상).
- **영향**: 실제 운영 설정으로 오해할 위험(예: "Supabase 연동돼 있겠지"라는 착각 → 문제 #1과 연결).
- **참고 방향**: 삭제하거나, 필요한 부분만 businesscard_qr 모듈로 이관.

### 8. 운영 기본값이 SQL 로그/show_sql 활성화
- **위치**: `businesscard_qr/src/main/resources/application.yml`
  ```yaml
  jpa.properties.hibernate.show_sql: ${SPRING_JPA_SHOW_SQL:true}   # 기본 true
  logging.level.org.hibernate.SQL: ${LOG_SQL_LEVEL:debug}          # 기본 debug
  logging.level.org.hibernate.orm.jdbc.bind: ${LOG_BIND_LEVEL:trace} # 기본 trace
  ```
- **내용**: 기본값이 SQL/바인딩 파라미터를 상세 출력. 별도 환경변수로 끄지 않으면 운영에서도 그대로.
- **영향**: 로그 과다, 성능 저하, **바인딩 파라미터에 개인정보(이메일/닉네임 등) 노출** 가능.
- **참고 방향**: 운영 기본값을 `false`/`warn`으로.

### 9. `view-count`가 소유자만 증가 가능 (조회수 의미와 모순)
- **위치**: `card/controller/BusinessCardController.java`(`/{cardId}/view-count`),
  `BusinessCardService.incrementViewCount`(`getOwnedActiveCard`)
- **내용**: 조회수 증가 API가 `/api/**`(인증 필요) + 소유자 카드만 대상. 즉 **명함 주인 본인만**
  조회수를 올릴 수 있음. 일반적으로 조회수는 "보는 사람"이 올리는 개념이라 설계 의도와 어긋날 수 있음.
- **영향**: 실제 방문자 조회수가 집계되지 않음(의도라면 무방).
- **참고 방향**: 조회수 증가를 공개 경로/토큰 기반 등으로 재설계할지 검토.
- **해결(2026-07-07)**: 소유자 제한 제거에 더해 `POST /view-count`를 **공개 경로**로 전환
  (`SecurityConfig` permitAll + `JwtAuthenticationFilter` 제외). 이제 QR을 스캔한 **비로그인 방문자**도
  조회수를 올릴 수 있어 "보는 사람이 올린다"는 의도가 실제로 실현된다. 서비스 시그니처도
  `incrementViewCount(cardId)`로 정리(userId 불필요).

### 10. 매 요청 DB 조회 + 리프레시 토큰 부재
- **위치**: `common/security/JwtAuthenticationFilter.java`(`userRepository.existsById`)
- **내용**: 인증된 모든 `/api/**` 요청마다 사용자 존재 여부를 DB로 확인 → 요청당 쿼리 1회.
  또한 리프레시 토큰이 없어 액세스 토큰(1시간) 만료 시 재로그인 필요.
- **영향**: 트래픽 증가 시 불필요한 DB 부하. UX상 잦은 재로그인.
- **참고 방향**: 캐싱 또는 토큰 클레임 신뢰 + 리프레시 토큰 도입 검토.

## ⚪ 낮음 / 정리

### 11. 예외 처리기가 한글 메시지 문자열로 상태코드 결정
- **위치**: `common/exception/GlobalExceptionHandler.java`
  (`determineStatusFromMessage`: `"찾을 수 없습니다"`, `"본인의"` 등 포함 여부로 404/403 판단)
- **내용**: "마이그레이션 용"이라 주석돼 있으나 문자열 매칭은 깨지기 쉬움(메시지 문구 변경 시 오동작).
- **참고 방향**: 커스텀 예외로 일원화 후 해당 분기 제거.

### 12. 게이트웨이 테스트 전면 비활성화
- **위치**: `dist_api_gateway/build.gradle`(`tasks.named('test') { enabled = false }`, test 소스셋 비움)
- **내용**: 프록시 핵심 로직(헤더 필터링, multipart 포워딩)에 대한 테스트가 전무.
- **참고 방향**: 최소한의 프록시 동작 테스트 추가.

### 13. 다운로드 토큰 단일 슬롯
- **위치**: `card/entity/BusinessCardEntity.java`(`issueVcfToken`/`issueImageToken`)
- **내용**: 토큰을 컬럼 하나에 덮어쓰기 저장 → 재발급 시 이전 URL 즉시 무효, 동시 다중 발급 불가.
- **참고 방향**: 사용 시나리오상 문제 없으면 유지, 필요 시 별도 토큰 테이블.

### 14. 업로드 콘텐츠 타입 미검증 / 업로드 초과 시 500
- **위치**: `BusinessCardService.storeImage`, `common/exception/GlobalExceptionHandler`
- **내용**: 이미지가 아닌 파일도 저장 가능(확장자만 정규화). 파일 크기 초과 시 전용 핸들러가 없어
  `MaxUploadSizeExceededException`이 일반 `Exception` 핸들러로 떨어져 **500** 반환(적절히는 413).
- **참고 방향**: 콘텐츠 타입 화이트리스트 + 업로드 초과 전용 핸들러.

### 15. 네이밍/기본값 정리
- **위치**: `businesscard_qr` 메인 클래스명 `DemoApplication`, 기본 JWT 시크릿이 소스에 하드코딩된
  로컬 기본값, OpenApiConfig의 `dev@example.com` 등.
- **내용**: 기능 문제는 아니나 프로젝트 정체성/운영 신뢰도 관점에서 정리 여지.
- **참고 방향**: 의미 있는 클래스명, 예시값 정리.
