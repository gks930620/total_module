## 전제: 이 문서를 읽을 때 감안할 것

- **total_module의 원래 컨셉**: 다른 곳에서 로컬 테스트를 전부 통과한 프로젝트를 가져와, 여기서는 **Railway 배포 기준으로 맞추는 작업만** 하는 저장소다 (요청 URL/컨트롤러 매핑, env, CORS 등).
- **그런데 businesscard_qr은 미완성 상태로 들어왔다.** 그래서 아래 문제들에는 "배포 적합화" 문제뿐 아니라 **기능 자체의 미완성 잔재**(앱의 타임아웃/401 처리 부재, 死코드, 커밋된 .env 등)가 섞여 있다. businesscard_qr에 한해서는 이 저장소에서 완성까지 진행하는 게 맞으며, 이는 예외적인 상황이다.
- **현재 브랜치(`buinsesscard_qr_app`) 작업 범위**: businesscard_qr이 미완성이므로 businesscard_qr + dist_api_gateway + 프로젝트 전반을 함께 손대도 됨. **프로젝트 A부터는** 로컬에서 완성 후 가져와 해당 모듈만 작업하는 원칙으로 복귀.

## 🔴 배포 직결 (반드시 처리)

### D1. `SPRING_DATASOURCE_URL` 미설정 시 조용히 H2 인메모리로 부팅 → 데이터 전부 소실
- **위치**: `businesscard_qr/src/main/resources/application.yml:11`
  ```yaml
  url: ${SPRING_DATASOURCE_URL:jdbc:h2:mem:businesscard;MODE=MySQL;...}
  ```
- **내용**: Railway에 배포했는데 `SPRING_DATASOURCE_URL` 환경변수를 빠뜨리면, **에러 없이** H2 인메모리 DB로 정상 부팅됩니다. 앱도 동작하는 것처럼 보이지만, **재시작·재배포할 때마다 회원/명함/이미지(`stored_files`)가 전부 초기화**됩니다. `APP_STORAGE_TYPE=db`로 이미지를 DB에 저장하도록 해놓은 노력이 무의미해지는 정확히 그 시나리오입니다.
- **`RailwayDeploymentValidator`는 JWT 시크릿만 검사**하고 datasource는 검사하지 않아 이 함정을 못 막습니다.
- **해결 방향**: `RailwayDeploymentValidator.validate()`에 "`RAILWAY_PROJECT_ID`가 있는데 최종 JDBC URL이 H2면 부팅 실패" 가드를 추가. + 배포 시 `SPRING_DATASOURCE_URL`은 **필수값**으로 문서화(→ [`현재프로젝트이후작업할거정리.md`](현재프로젝트이후작업할거정리.md) Railway 섹션).

### D2. QR·이미지·다운로드 절대 URL이 X-Forwarded 헤더에 의존 (게이트웨이 뒤에서 깨질 수 있음)
- **위치**: `card/service/BusinessCardService.java`의 URL 빌드부(다운로드 URL 발급, `toResponse`의 `imageUrl`) — `ServletUriComponentsBuilder.fromCurrentContextPath()` 사용. `application.yml:3` `forward-headers-strategy: framework`.
- **내용**: 서버가 QR/이미지/다운로드 **절대 URL을 "현재 요청의 호스트" 기준으로 생성**합니다. 클라이언트는 게이트웨이(`distapigateway-...`)를 바라보는데, 게이트웨이 → businesscard_qr 중계 시 `X-Forwarded-Host/Proto/Prefix`가 제대로 전달·반영되지 않으면 생성된 URL이 **businesscard_qr의 내부 호스트나 `http://`** 로 찍힙니다.
  - 결과: QR 코드를 폰 카메라로 찍어도 접근 불가, 명함 이미지 `Image.network` 404.
  - 앱은 서버가 준 URL을 그대로 쓰므로(정상 동작) **보정이 불가능** → 반드시 서버/게이트웨이 쪽에서 맞아야 함.
- **교차 확인**: 앱 리뷰(#3), 백엔드 리뷰(M7), 게이트웨이 리뷰(#2, Location 미재작성)에서 모두 지적된 **모듈 경계 문제**.
- **해결 방향**:
  1. (권장) 공개 base URL을 `APP_PUBLIC_BASE_URL` 같은 **환경변수로 명시**해 요청 헤더에서 유추하지 않도록. 가장 확실.
  2. 또는 게이트웨이가 `X-Forwarded-Host/Proto/Prefix`를 정확히 전달하는지 검증 + businesscard_qr가 이를 신뢰하도록 유지.
- **필수**: 배포 후 실기기에서 `business_card_image_url`과 QR `url` 값이 **`https://distapigateway-...`로 시작하는지 눈으로 확인**.

### D3. 앱에 HTTP 타임아웃이 전혀 없음 → 콜드스타트 시 무한 로딩
- **위치**: `businesscard_qr_app/lib/services/business_card_service.dart` 전 호출부(get/post/delete/`request.send()`), `lib/services/image_cache_service.dart`.
- **내용**: 모든 네트워크 호출에 `.timeout(...)`이 없습니다. Railway 무료/hobby 티어는 유휴 시 서비스를 재우고, **깨어나는 첫 요청이 수 초~수십 초 지연**되거나 멈출 수 있습니다. 타임아웃이 없으면 Future가 영영 완료되지 않아 로그인 흐름과 목록 로딩 스피너(`main.dart`)가 **무한 회전**하고 사용자는 앱을 강제 종료하는 수밖에 없습니다.
- **해결 방향**: 각 호출을 `.timeout(Duration(seconds: 15~30))`으로 감싸고, 실패 시 **재시도 가능한 에러 UI**로 표면화.

### D4. 앱이 401·5xx·토큰 재발급을 처리하지 않음 → 서버 재배포 후 계속 실패
- **위치**: `business_card_service.dart` (백엔드 JWT를 `static` 필드에 캐시, 로컬 시계로만 만료 추정; `_extractData`는 모든 응답을 JSON으로 가정).
- **내용**:
  - 서버가 실제 **401**을 돌려줘도 캐시된 토큰을 무효화하거나 재발급하지 않음. Railway 재배포로 **JWT 시크릿/서명이 바뀌면** 기존 토큰이 거부되어 이후 모든 호출이 실패하는데 복구 경로가 없음(앱 강제 종료 전까지).
  - 백엔드는 `refreshToken`과 `POST /api/auth/refresh`를 제공하는데 **앱은 둘 다 무시**하고 JWT를 영구 저장(`shared_preferences` 미사용)하지도 않음.
  - `_extractData`가 **모든 응답 본문을 JSON으로 가정** → Railway/게이트웨이가 콜드스타트·배포 중 내보내는 **HTML/텍스트 502·503·504**를 만나면 "유효한 JSON이 아님"으로 뭉뚱그려지고 재시도 경로도 없음.
- **해결 방향**: `401` 수신 시 캐시 토큰 clear → `refresh` 1회 시도 후 재요청; 상태코드/Content-Type로 5xx를 분기해 "서버 깨우는 중, 재시도" 경로 제공.

## 🟡 중간 (실사용 중 오동작 가능)

### G1. 게이트웨이가 업스트림 다운 시 500 반환 (502/503/504가 맞음)
- **위치**: `gateway/controller/ApiGatewayController.java`의 `forward` (try/catch·`@ControllerAdvice` 없음).
- **내용**: 업스트림(`businesscard_qr`)이 연결 거부/타임아웃일 때 `RestTemplate.execute`가 `ResourceAccessException`을 던지고, 처리기가 없어 Spring 기본 에러 컨트롤러가 **500**을 반환. Railway는 배포마다 업스트림이 잠깐 내려가므로 그 구간에 클라이언트가 "게이트웨이 버그"처럼 보이는 500을 받음. 스트리밍 중 실패 시 이미 200 커밋된 상태로 본문이 잘릴 수도 있음.
- **해결 방향**: `ResourceAccessException`/`IOException`을 잡아 connect/timeout을 502/504로 매핑. 200 경로 status는 헤더 확보 후에만 설정.

### B1. 전역 `Exception` 핸들러가 표준 Spring MVC 4xx를 500으로 뒤바꿈
- **위치**: `common/exception/GlobalExceptionHandler.java` (catch-all `@ExceptionHandler(Exception.class)`).
- **내용**: `ExceptionHandlerExceptionResolver`가 프레임워크 기본 리졸버보다 먼저 도는데 catch-all이 있어, 명시 처리되지 않은 표준 예외가 전부 500이 됨:
  - `payload` 파트 누락(`MissingServletRequestPartException`) → 400이어야 하나 **500**
  - 비-multipart 본문(`HttpMediaTypeNotSupportedException`) → 415 → **500**
  - 잘못된 메서드(`HttpRequestMethodNotSupportedException`) → 405 → **500**
  - 없는 경로(`NoResourceFoundException`) → 404 → **500**
- **영향**: 클라이언트 실수가 서버 오류로 표시되고, 로그/모니터링에서 진짜 500이 묻힘.
- **해결 방향**: `GlobalExceptionHandler extends ResponseEntityExceptionHandler`로 변경하거나, `org.springframework.web.*` 예외는 catch-all에서 제외.

### B2. 카카오 외부 HTTP 호출이 쓰기 트랜잭션 안에서 실행
- **위치**: `auth/service/AuthService.java` `loginWithKakaoAccessToken`(`@Transactional`)이 DB 작업 전에 `fetchKakaoUserProfile()`(10s 타임아웃 블로킹 HTTP)를 호출.
- **내용**: 원격 호출 동안 트랜잭션/영속성 컨텍스트가 열린 채 유지 → 카카오 지연·장애 시 트랜잭션 자원과 커넥션 풀이 묶임.
- **해결 방향**: 카카오 프로필 조회는 트랜잭션 밖에서, `syncKakaoUser + issueTokens`만 `@Transactional`로.

### B3. `driver-class-name` 기본값이 `org.h2.Driver`로 고정
- **위치**: `businesscard_qr/application.yml:12`.
- **내용**: 명시 기본 드라이버가 Boot의 URL 기반 자동감지를 덮어써서, 운영자가 `SPRING_DATASOURCE_URL=jdbc:mariadb://...`만 넣고 `SPRING_DATASOURCE_DRIVER`를 빠뜨리면 "드라이버가 URL을 이해 못 함"으로 **부팅 실패**.
- **해결 방향**: `driver-class-name` 제거(Boot가 URL로 추론)하거나 기본값을 MariaDB 드라이버로.

### B4. 조회수 증가가 비원자적 read-modify-write
- **위치**: `BusinessCardService.incrementViewCount` + `BusinessCardEntity.incrementViewCount()`.
- **내용**: 엔티티 로드 후 `viewCount += 1`. 비인증 공개 엔드포인트라 다수 QR 스캐너가 동시 접근 → **lost update**(과소 집계).
- **해결 방향**: 리포지토리에서 `UPDATE business_cards SET view_count = view_count + 1 WHERE id = ? AND is_active = true` 원자적 쿼리 사용.

### A1. 앱 이미지 업로드에 Content-Type 미지정 → 실기기 업로드 실패
- **위치**: `business_card_service.dart:67` `MultipartFile.fromPath('businessCardImage', path)` (contentType 없음).
- **내용**: 백엔드는 `Content-Type`이 `image/`로 시작 + 확장자 `{jpg,jpeg,png,gif,bmp,webp}`를 **강제**. `http` 패키지가 mime 추론 실패 시 `application/octet-stream` → 400. iOS `image_picker`는 `.heic`를 줄 수 있어 허용 목록 밖 → 400. **실기기에서만** 재현.
- **해결 방향**: `contentType: MediaType('image', ext)` 명시 + picker 출력을 jpg/png로 제약/변환.

### G2. 게이트웨이 리다이렉트 처리 불일치 + 내부 URL 유출
- **위치**: `GatewayClientConfig.java`(`SimpleClientHttpRequestFactory`, 리다이렉트 오버라이드 없음), `ApiGatewayController.filterResponseHeaders`(`Location` 미재작성).
- **내용**: GET 리다이렉트는 게이트웨이가 자동 추종(클라이언트가 3xx를 못 봄), POST 등은 미추종이라 업스트림 `Location`(내부 절대 URL)이 그대로 전달됨 → `Location` 재작성이 없어 **내부 호스트 유출** + 클라이언트 깨짐. swagger `/swagger-ui.html`의 302, 트레일링 슬래시 리다이렉트에서 발생.
- **해결 방향**: 자동추종 일괄 비활성화 + `Location`(및 `Set-Cookie` Domain)을 업스트림 base → 게이트웨이 오리진으로 재작성.

## ⚪ 낮음 / 정리·개선

### 백엔드
- **B5. 이미지 "다운로드 토큰"이 무력화됨**: `/uploads/**`가 인증·토큰 없이 공개 서빙되고 `toResponse`가 그 공개 URL을 `imageUrl`로 내려줌 → 5분 토큰 엔드포인트는 실효성 없음(파일명 추측 난이도에만 의존). *이미지 공개가 최종 결정이면* 토큰 세리머니를 제거해 "보호되는 척"을 없애는 게 정직함. (설계상 이미 인지된 항목)
- **B6. 클라이언트 지정 card id의 TOCTOU**: `existsById` 후 `save()`(merge)라 동일 UUID 동시 생성 시 나중 요청이 기존 행을 UPDATE(소유자 덮어쓰기) 가능. 창이 좁지만 DB 유니크/insert 가드가 없음 → `persist` 경로로.
- **B7. 카카오 토큰의 app(audience) 미검증**: `/v2/user/me`로 resolve만 되면 통과 → 다른 카카오 앱 토큰도 계정 프로비저닝. 저위험이나 확인 권장.
- **B8. `IllegalStateException`을 일괄 409로 매핑**: 내부 오류도 409로 표시되어 오해 소지. "이미 삭제/중복"에만 409, 나머지는 500으로.
- **B9. 리프레시 TTL이 env 미설정 + 진짜 폐기 아님**: `JwtTokenProvider`가 14일을 생성자 기본값으로만; `refresh`가 구 토큰을 무효화 안 함(무상태) → "회전(rotation)" 표현은 과장.
- **운영 주의(코드 문제 아님)**: `APP_STORAGE_TYPE=db`로 10MB 이미지를 저장하려면 Railway MariaDB의 `max_allowed_packet`이 충분히 커야 함.

### 게이트웨이
- **G3. 테스트가 실제 HTTP 경로를 우회**: `MockRestServiceServer`가 `SimpleClientHttpRequestFactory`(스트리밍/타임아웃/리다이렉트/멀티파트)를 대체 → 핵심 가치(스트리밍·raw 멀티파트 포워딩·업스트림 다운)가 **미검증**. MockWebServer 등 와이어 레벨 테스트 필요.
- **G4. `URI.create`가 원본 쿼리로 500 가능**: 인코딩 안 된 예약문자(공백/`{`/`|` 등) 쿼리에서 `IllegalArgumentException` → 500. 관대한 프록시라면 통과시켜야.
- **G5. 블로킹·비풀 프록시**: `SimpleClientHttpRequestFactory`(커넥션 풀 없음, 60s read timeout) + 동기 처리 → 업스트림 지연 시 Tomcat 200스레드 고갈, `/healthz`까지 영향 → Railway 헬스체크 실패·재시작 루프 위험. 풀 클라이언트(Apache HttpClient) 검토.

### 앱(Flutter)
- **A2. `usesCleartextTraffic="true"`**: HTTPS엔 불필요하고 차단도 안 하지만, 실수로 `http://` URL(→ D2)이 조용히 통과. URL이 HTTPS로 확정되면 제거 권장. (`AndroidManifest.xml`)
- **A3. 시작 크래시 위험 + 커밋된 비밀**: `main.dart`가 `dotenv.env['KAKAO_NATIVE_APP_KEY']!` 강제 언랩(키 없으면 UI 전에 크래시), 카카오 네이티브 키가 `.env`에 커밋 + 에셋 번들.
- **A4. async 경계에서 `mounted` 가드 없이 `context` 사용**: 느린(→D3) 호출 중 화면 이탈 시 `ScaffoldMessenger.of(context)` 등이 throw 가능.
- **A5. 死코드/중복**: `_loginWithKakao`/`_syncUserToBackend`가 두 화면에 그대로 중복(둘째는 死코드); `incrementViewCount`·`downloadImageToGallery`가 UI에서 호출 안 됨.
