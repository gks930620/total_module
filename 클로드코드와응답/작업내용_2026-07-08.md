## 2. businesscard_qr (백엔드) — 테스트 22/22 ✅

### D1 — H2 인메모리 부팅 차단 (fail-fast)
`common/config/RailwayDeploymentValidator.java`
`RAILWAY_PROJECT_ID`가 있는 환경에서 `spring.datasource.url`이 비었거나 `jdbc:h2`로 시작하면 기존 JWT 시크릿 검사와 같은 방식으로 **부팅을 실패**시킴. "재배포마다 데이터가 사라진다"는 한글 에러 메시지 포함.

### D2 — 공개 base URL 환경변수화
- `application.yml`: `app.public-base-url: ${APP_PUBLIC_BASE_URL:}` 신설
- 신규 `card/support/PublicUrlResolver.java`: 설정돼 있으면 그 값(트레일링 슬래시 제거), 없으면 기존 `ServletUriComponentsBuilder` 폴백
- `BusinessCardService`의 절대 URL 생성 4곳(vcf/이미지 다운로드 URL, `toResponse`의 imageUrl/vcfUrl) 전부 교체
- Railway인데 미설정이면 **WARN 로그** (기존 배포가 X-Forwarded로 동작 중일 수 있어 하드 실패는 안 함)
- ⚠️ **Railway에 `APP_PUBLIC_BASE_URL=https://<게이트웨이 공개 도메인>` 추가 권장** → [`현재프로젝트이후작업할거정리.md`](현재프로젝트이후작업할거정리.md) B2 표 참고

### B1 — 예외 상태코드 정상화
`GlobalExceptionHandler`가 `ResponseEntityExceptionHandler` 상속으로 변경. 없는 경로 404, 잘못된 메서드 405, `payload` 파트 누락 400, 비-multipart 415, 업로드 초과 413 — 전부 기존 `{success:false, message, errorCode}` 봉투 유지.

### B2 / B3 / B4 / B6 / B8 / B9 / B7
- **B2**: `loginWithKakaoAccessToken`/`refresh`의 `@Transactional` 제거. DB 작업은 별도 빈 `UserService.syncKakaoUser`(자체 `@Transactional`)에서만 — 카카오 지연이 커넥션 풀을 묶지 않음
- **B3**: `driver-class-name: ${SPRING_DATASOURCE_DRIVER:}` — 빈 값이면 Boot가 JDBC URL로 드라이버 추론 (MariaDB URL만 넣어도 부팅 OK)
- **B4**: `BusinessCardRepository.incrementViewCount` — `UPDATE ... SET view_count = view_count + 1 WHERE id=? AND is_active=true` 원자 쿼리. 0건이면 기존과 동일한 404
- **B6**: `BusinessCardEntity`가 `Persistable<String>` 구현 (`@Transient isNew` + `@PostLoad @PostPersist`) — 신규 생성은 진짜 INSERT라 동시 중복 시 PK 위반 → 기존 409 핸들러로 연결. 수정/삭제 경로는 기존과 동일
- **B8**: `IllegalStateException`→409 핸들러 제거. 요청 경로에서 던지는 곳이 없음을 확인(스타트업 검증기뿐), 의도적 충돌은 이미 `DuplicateResourceException`(409) 사용 중이라 새 예외 불필요
- **B9**: `APP_JWT_REFRESH_TOKEN_VALIDITY_SECONDS` (기본 1209600 = 14일)
- **B7**: `APP_KAKAO_EXPECTED_APP_ID` 설정 시 `access_token_info`로 토큰의 `app_id` 검증. **미설정(기본)이면 동작 변화 0**

### 신규 테스트 (ApiIntegrationTest에 3건 추가)
없는 경로 404 / 잘못된 메서드 405 / payload 누락 400 — B1 회귀 방지.

## 3. dist_api_gateway (게이트웨이) — 테스트 16/16 ✅

### G5 — Apache HttpClient5 풀링 전환 (다른 픽스의 기반)
`build.gradle` + `GatewayClientConfig.java`
- `PoolingHttpClientConnectionManager` (총 100 / 라우트당 50), connect 5s, read/response **30s** (60s에서 축소 — Railway 엣지 타임아웃보다 짧게)
- `.disableRedirectHandling()` (G2), `.disableContentCompression()` (gzip 원본 그대로 중계 불변식 유지), `.disableAutomaticRetries()` (프록시가 비멱등 요청을 몰래 재시도하면 안 됨)
- ⚠️ **중요 발견**: Spring 6.1의 HttpComponents 팩토리는 `getBody()` 사용 시 **본문 전체를 버퍼링**함 — 그대로 뒀으면 대용량 업로드 OOM 방지 불변식이 조용히 깨졌음. `StreamingHttpOutputMessage.setBody(...)`로 진짜 스트리밍 유지

### G1 — 업스트림 장애 시 올바른 상태코드
connect 실패/거부 → **502**, read 타임아웃 → **504**, 본문 `{"success":false,"message":"업스트림 서비스에 연결할 수 없습니다"}`. 스트리밍 도중 실패로 응답이 이미 커밋됐으면 로그만 남기고 중단(잘린 200에 덮어쓰기 시도 안 함).

### G2 — 리다이렉트 일원화 + Location 재작성
모든 메서드에서 3xx를 그대로 중계하되, `Location`이 업스트림 base URL로 시작하면 게이트웨이 공개 오리진으로 재작성(경계 문자 검사로 `:80` vs `:8081` 오탐 방지). 상대 Location은 그대로 통과. → 내부 호스트 유출 차단.

### G4 — 쿼리스트링 관대 처리
`URI.create` 실패 시 문제 문자만( 공백 `"` `<` `>` `{` `}` `|` `\` `^` `` ` `` ) 인코딩해 1회 재시도, 그래도 실패면 400. 정상 경로는 기존 그대로 zero-copy.

### G3 — 와이어 레벨 테스트 신규 13건
MockWebServer 기반 — 기존 MockRestServiceServer 테스트가 우회하던 실제 HTTP 경로 검증:
- 1MB 멀티파트 **바이트 동일** 전달(boundary 보존) / 404 JSON passthrough / 302 Location 재작성 / 상대 Location 통과 / X-Forwarded 헤더 도달 / 인코딩된 쿼리 통과 / 512KB 바이너리 응답 동일 / **업스트림 다운 → 502** / G4 인코더 단위 5건
- 테스트용 `application.yml`에 운영과 동일한 `multipart.enabled=false` + `forward-headers-strategy` 반영 (이게 없으면 테스트가 운영과 다른 조건으로 돎)

## 4. businesscard_qr_app (Flutter) — `flutter analyze` 오류 0 ✅

### D3 — 전 호출 타임아웃
일반 20s / 업로드·이미지 60s. `TimeoutException` → `'서버 응답이 없습니다. 잠시 후 다시 시도해주세요.'`

### D4 — 401 복구 + refresh + 5xx 분기
- `refreshToken` 저장 시작 (기존엔 버렸음), 백엔드 `POST /api/auth/refresh` 연동 (요청 필드명 `refreshToken` — 서버 DTO 확인함)
- `_sendWithAuthRetry`: **401 → 토큰 클리어 → refresh 시도 → 실패 시 카카오 재로그인 → 원요청 1회 재시도** (무한루프 가드). 인증 필요한 7개 호출 전부 이 경로로 리팩터. 멀티파트는 재전송 불가라 시도마다 재구성
- `_extractData`: 502/503/504 → `'서버가 깨어나는 중입니다...'`, 비-JSON → `'서버 오류 (코드)'`. `FormatException` 유출 불가

### A1 — 업로드 Content-Type
확장자→`MediaType('image', ...)` 명시 (jpg/jpeg/png/gif/bmp/webp), 그 외(.heic 등)는 업로드 전에 한글 에러. `pickImage`에 `imageQuality: 85` — iOS에서 HEIC가 JPEG로 재인코딩되어 애초에 도달 안 함. `http_parser` 의존성 추가(pub get 완료).

### A2 / A3 / A4 / A5
- **A2**: `AndroidManifest.xml`에서 `usesCleartextTraffic` 제거
- **A3**: `KAKAO_NATIVE_APP_KEY` null-safe 읽기, 없으면 크래시 대신 안내 화면(`ConfigErrorApp`)
- **A4**: async 경계 전수 조사 — `mounted`/`context.mounted` 가드 추가, `use_build_context_synchronously` 린트 0
- **A5**: 리스트 화면의 중복 `_loginWithKakao`/`_syncUserToBackend` 삭제(~60줄, 상호참조만 있던 死코드 확인). `incrementViewCount`/`downloadImageToGallery`는 API 유지 + `// TODO: UI 연결 예정`
