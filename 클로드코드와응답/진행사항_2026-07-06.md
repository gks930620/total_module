## 1. 핵심 완성: 이미지 영구 저장 (Railway MySQL)

**문제(#1)**: 이미지를 컨테이너 로컬 디스크(`./uploads`)에 저장 → Railway 재배포 시 소실.
원래 외부 스토리지로 가려다 미완성 상태였음.

**해결 방향(정정 반영)**: 외부 오브젝트 스토리지(Supabase)가 아니라 **Railway MySQL에 이미지 바이트를
저장**하여 별도 인프라 없이 영속성 확보.

**구현 — 스토리지 추상화 도입** (`card/storage` 패키지 신설)
- `FileStorage` 인터페이스 + `StoredFile` 레코드
- `DbFileStorage` — `stored_files` 테이블(`LONGBLOB`)에 저장 **(운영 기본값)**
  - `StoredFileEntity`, `StoredFileRepository`
- `LocalFileStorage` — 로컬 디스크 (개발 편의용, `app.storage.type=local`)
- `FileStorageConfig` — `app.storage.type` 값으로 구현 선택 (기본 `db`)
- `FileController` — `/uploads/**` 를 스토리지 경유로 서빙(정적 리소스 핸들러 대체).
  저장 백엔드가 무엇이든 이미지 인라인 표시가 동일하게 동작.
- `BusinessCardService` 리팩터: 저장/삭제/다운로드/vCard 사진을 모두 `FileStorage`로 위임.
  논리 경로 규칙(`/uploads/business-card-images/<파일명>`)은 그대로 유지 → **DB/응답 계약 불변**.

**설정**: `APP_STORAGE_TYPE` (기본 `db`). 로컬 디스크가 필요하면 `local`.

> 참고: 통합 테스트도 실제 운영 경로(DB 저장)를 그대로 사용하도록 수정하여 회귀를 검증한다.

## 5. API 완성 및 검증 (#9, #10, #14)

- **업로드 검증 (#14)**: `storeImage`에서 콘텐츠 타입(`image/*`) + 확장자 화이트리스트 검증.
- **413 핸들러 (#14)**: `MaxUploadSizeExceededException` → `413 PAYLOAD_TOO_LARGE`
  (기존엔 500으로 떨어짐).
- **조회수 (#9)**: `incrementViewCount`의 소유자 제한 제거 → 인증된 사용자가 활성 카드의
  조회수를 증가시킬 수 있도록 변경(조회수 의미에 부합).
- **리프레시 토큰 (#10 완성)**: 액세스 토큰만 있고 재로그인만 가능하던 구조를 보완.
  - `JwtTokenProvider`: refresh 토큰 발급/타입 검증(`typ=refresh`), 유효기간(기본 14일).
  - `AuthTokenResponse`에 `refreshToken` 필드 추가.
  - `POST /api/auth/refresh` 신설 — 유효한 refresh 토큰 + 실제 사용자 확인 후 **재발급**.
    (정정: 무상태 JWT라 구 refresh 토큰이 폐기되지 않으므로 진짜 "회전"은 아니다. 상세는 프로젝트_개요 §4.2 참고.)
  - 단위 테스트로 access/refresh 구분 검증.

## 7. 이번에 "의도적으로 남겨둔" 항목 (근거 포함)

정직하게 밝히면, 아래는 이번 범위에서 코드 변경 대신 **현행 유지 + 근거 기록**으로 처리했습니다.

- **#3 `/uploads/**` 공개**: 명함 이미지는 QR로 공유되는 특성상 공개 열람이 필요.
  파일명이 UUID 기반이라 추측이 어렵고, 이제 서빙 경로가 `FileController`로 일원화되어
  향후 인증/서명 URL을 얹기 쉬운 구조가 됨. (현재는 공개 유지가 의도)
- **#10 매 요청 `existsById` 조회**: **삭제된** 사용자의 토큰을 즉시 무효화하기 위한
  **보안상 의도된 동작**으로 유지. (리프레시 토큰은 별도로 추가 완료)
  (정정: 근거를 "삭제된 사용자"로 축소한다. `UserEntity`에 차단(block) 상태 필드가 없어
  현재 무효화되는 경우는 사용자 행 삭제뿐이다. 차단 기능은 로드맵 항목.)
- **#11 예외 핸들러의 한글 문자열 매칭**: 정상 동작하며 위험도 낮음. 커스텀 예외 일원화는
  후속 리팩터 과제로 남김.
- **#13 다운로드 토큰 단일 슬롯**: 현재 사용 시나리오에서 문제 없어 유지.
