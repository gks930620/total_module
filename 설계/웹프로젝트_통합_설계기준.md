# 웹 프로젝트 통합 설계기준 — total_module에 프로젝트 A/B(웹)를 넣을 때

> 작성일: 2026-07-08 (같은 날 §3 확정 반영: 게이트웨이 경유 폐기 → 전 프로젝트 자기 URL 직행)
> 대상: 앱이 아니라 **웹 화면이 있는 프로젝트**를 total_module로 가져올 때의 기준.
> 방향 기준: [`아키텍처_방향결정.md`](아키텍처_방향결정.md) / 구조 원칙: [`프로젝트구조_설계원칙.md`](프로젝트구조_설계원칙.md)

## 0. 먼저 판단할 것: 프로젝트가 어느 유형인가?

| 유형 | 화면(HTML/JS)은 어디서 나오나 | API는 |
|------|------------------------------|-------|
| **유형 1. 정적웹 분리형** | 별도 정적 호스팅 (github.io, Vercel 등) | total_module의 스프링부트 모듈 |
| **유형 2. 스프링 일체형** | 스프링부트가 직접 서빙 (@Controller + 템플릿/정적파일) | 같은 스프링부트의 @RestController |

> 클라이언트가 **네이티브 앱**인 경우(명함QR처럼)는 유형 1과 동일하게 취급하되 CORS가 필요 없다.

---

## 1. 유형 1 — 정적웹 분리형 (CSR + API 분리 배포)

```
[정적 호스팅 (github.io 등)]      ← HTML/JS/CSS. total_module과 무관하게 별도 배포
        │  브라우저 JS가 fetch("https://project-a-xxx.up.railway.app/api/...")
        ▼
[projectA 모듈 (백엔드만)]        ← 자기 공개 URL로 직접 받음 (게이트웨이 없음)
```

### 옮기는 것
- **백엔드 모듈 전체** (컨트롤러만이 아니라 서비스·엔티티·build.gradle 포함 폴더째).
  "REST API만 옮긴다"는 역할 얘기고, 물리적으로는 스프링부트 프로젝트 전체가 모듈로 들어온다.

### 옮기면서 빼는 것
- 스프링이 화면을 서빙하던 부분(템플릿, `static/`의 화면용 리소스) — 화면은 정적 호스팅으로 갔으므로 제거.
  (dist_api_gateway 최소화 때 templates/ 잔재를 지운 것과 같은 정리)

### 옮기면서 고치는 것 (Railway 적응 = 이 저장소의 본업)
1. **CORS** ← 웹이라 필수. 정적 호스팅 오리진(`https://xxx.github.io`) 허용 + OPTIONS preflight 응답 확인
2. **인증 방식 확인** ⚠️ 최우선 점검 항목:
   - **JWT 토큰 방식** → 그대로 OK
   - **세션+쿠키 방식** → 교차 도메인(github.io ↔ railway.app)에서 SameSite 문제로 쿠키가 안 붙음 → 토큰 방식으로 전환하거나 상당한 손질 필요
3. Railway 기본기: 설정 전부 env 주입, `GET /healthz`, 로컬 디스크 저장 금지, `PORT` env

---

## 2. 유형 2 — 스프링 일체형 (화면+API 한 프로젝트)

```
[브라우저] ──▶ [projectA 모듈 하나]
                 ├ @Controller      → 화면(HTML) 반환
                 └ @RestController  → API(JSON) 반환
```

스프링부트 하나가 화면과 API를 둘 다 서빙하는 평범한 구성. **모듈 하나로 통째로 넣으면 그대로 동작한다.**

### 이 방식의 장점 — 문제가 오히려 사라짐
화면과 API가 **같은 도메인**에서 나오므로:
- **CORS 불필요** (교차 출처가 아님)
- **세션+쿠키 로그인도 그냥 됨** (SameSite 문제 없음)
- 배포 관리 한 곳

---

## 3. 접근 방식: 전 프로젝트 자기 공개 URL 직행 (확정)

**2026-07-08 확정** ([`아키텍처_방향결정.md`](아키텍처_방향결정.md)): 게이트웨이 경유 방식은 폐기.
모든 프로젝트는 Railway 서비스의 **자기 공개 도메인으로 직접** 트래픽을 받는다.

- 경로 프리픽스(`/a/**`) 작업 불필요 — 컨트롤러/정적 리소스 매핑을 손댈 이유가 없음
- 프로젝트가 거의 그대로 들어감 (특히 유형 2는 정적 리소스 절대경로 문제 자체가 발생 안 함)
- 도메인이 서비스마다 따로 생기는 건 감수 (필요하면 나중에 커스텀 서브도메인 `a.mydomain.com`으로 정리 가능)

**모듈 편입 절차**: 폴더 편입 → settings.gradle include → Dockerfile COPY/bootJar → start.sh APP_MODULE 목록 → Railway 서비스 생성(`APP_MODULE=project_a`) + Generate Domain. (상세: [`아키텍처_방향결정.md`](아키텍처_방향결정.md) §3)

> 참고(폐기 근거): 게이트웨이 경유는 "도메인 하나로 통일"이 유일한 장점인데, 화면·API·정적리소스 전부
> `/a` 프리픽스 공사 + 게이트웨이 다중 라우팅 구현이 필요해서 비용이 훨씬 컸다.

