> 작성일: 2026-07-06
> 대상 커밋: `fee488a` (브랜치 `buinsesscard_qr_app`)
> 이 문서는 "이 프로젝트가 무엇을 하는가"를 정리한 것입니다. 문제점은 `클로드코드와응답/문제점_정리.md` 참고.

---

## 1. 한 줄 요약

카카오 로그인으로 인증한 사용자가 **디지털 명함(business card)을 등록/조회/수정/삭제**하고,
명함을 **vCard(.vcf) 파일 또는 이미지로 다운로드**할 수 있게 해주는 백엔드 서비스.
"개발은 모듈로 분리, 배포는 하나의 Docker 이미지로 통합"하는 **모듈러 모놀리스** 구조.

클라이언트는 별도 저장소의 **Flutter 앱**(이 저장소에는 포함되지 않음).

---

## 2. 리포지토리 구조 (멀티 모듈 Gradle)

```
total_module/                     # 루트 (Gradle 멀티모듈 + 배포 컨테이너)
├── Dockerfile                    # 멀티스테이지 빌드. 두 모듈 jar을 모두 빌드
├── railway.toml                  # Railway 배포 설정 (healthcheck /healthz)
├── settings.gradle               # include 'businesscard_qr', 'dist_api_gateway'
├── build.gradle                  # 루트 (거의 비어있음)
│
├── businesscard_qr/              # [핵심] 명함 API 서버 (Spring Boot)
├── dist_api_gateway/             # [게이트웨이] 얇은 리버스 프록시 (Spring Boot)
│
├── uploads/                      # 업로드된 명함 이미지 저장 (로컬 디스크, gitignore)
└── 설계/                          # 설계 문서 폴더
```

- **배포 방식**: 루트에 단일 `Dockerfile`을 두고, Railway 서비스마다 환경변수
  `APP_MODULE`(`businesscard_qr` 또는 `dist_api_gateway`)로 실행할 jar을 선택.
- 두 모듈 모두 `GET /healthz` 를 노출하여 Railway healthcheck에 대응.

## 4. 모듈 ①: `businesscard_qr` (핵심 API 서버)

명함 도메인의 실제 비즈니스 로직을 담당. 기본 포트 `8081`.

### 4.1 패키지 구조 (도메인별)

```
com.businesscard
├── auth/      카카오 로그인 → JWT 발급
├── user/      사용자 동기화/저장 (app_users 테이블)
├── card/      명함 CRUD, 이미지 저장, vCard/이미지 다운로드
└── common/    보안(JWT 필터), 예외 처리, CORS/정적리소스 설정, 공통 응답 DTO
```

### 4.2 인증 흐름

1. Flutter 앱이 카카오 SDK로 카카오 액세스 토큰 획득.
2. `POST /api/auth/kakao` 로 카카오 액세스 토큰 전달.
3. 서버가 `https://kapi.kakao.com/v2/user/me` 를 호출해 토큰 검증 + 프로필 조회.
4. 카카오 ID로 사용자 upsert (`id = "kakao_" + kakaoId`, 테이블 `app_users`).
5. 자체 **JWT 액세스 토큰**(유효기간 기본 1시간) 발급하여 응답.
6. 이후 요청은 `Authorization: Bearer <jwt>` 헤더로 인증.
   - `JwtAuthenticationFilter`가 토큰 검증 + `userRepository.existsById`로 사용자 존재 확인 후
     `SecurityContext`에 인증 정보 설정.

> 리프레시 토큰 지원(2026-07-06 추가): `POST /api/auth/refresh` 로 액세스/리프레시 토큰 **재발급**.
> ⚠️ 리프레시 토큰은 무상태 JWT이며 서버 저장/블랙리스트가 없어 기존 토큰이 만료(기본 14일)까지
> 그대로 유효하다. 즉 진짜 "회전(rotation, 구 토큰 무효화)"은 아니고 재발급이다.
> 진짜 회전이 필요하면 refresh 토큰 저장소(jti 화이트리스트/버전 컬럼) 도입 필요.

### 4.3 주요 API 엔드포인트

| 메서드 | 경로 | 인증 | 설명 |
|--------|------|------|------|
| POST | `/api/auth/kakao` | 공개 | 카카오 로그인 → JWT(액세스+리프레시) 발급 |
| POST | `/api/auth/refresh` | 공개 | 리프레시 토큰으로 재발급(무상태 JWT — 구 토큰 무효화 없음) |
| POST | `/api/users/sync` | 필요 | 사용자 프로필 동기화 |
| GET | `/api/business-cards` | 필요 | 내 명함 목록 |
| GET | `/api/business-cards/{id}` | 필요 | 명함 단건 조회 |
| POST | `/api/business-cards` | 필요 | 명함 생성 (multipart: payload JSON + 이미지) |
| PUT | `/api/business-cards/{id}` | 필요 | 명함 수정 (multipart) |
| DELETE | `/api/business-cards/{id}` | 필요 | 명함 소프트 삭제 (`is_active=false`) |
| POST | `/api/business-cards/{id}/view-count` | **공개** | 조회수 증가 (비로그인 방문자도 집계) |
| GET | `/api/business-cards/{id}/vcf-download-url` | 필요 | vCard 다운로드용 임시 URL 발급 |
| GET | `/api/business-cards/{id}/image-download-url` | 필요 | 이미지 다운로드용 임시 URL 발급 |
| GET | `/api/business-cards/{id}/downloads/vcf?token=` | **공개** | vCard 파일 다운로드 (토큰 검증) |
| GET | `/api/business-cards/{id}/downloads/image?token=` | **공개** | 이미지 파일 다운로드 (토큰 검증) |
| GET | `/healthz` | 공개 | 헬스체크 |
| GET | `/swagger-ui.html`, `/v3/api-docs/**` | 공개 | API 문서 |

### 4.4 명함 다운로드(토큰) 메커니즘

- 다운로드 URL 발급 API가 **UUID 토큰 + 5분 만료시각**을 엔티티에 저장하고, 토큰이 붙은 URL을 반환.
- 실제 다운로드 엔드포인트(`/downloads/vcf`, `/downloads/image`)는 **인증 없이** 토큰만으로 접근 가능
  (앱 외부 브라우저/공유 상황을 고려한 설계).
- vCard는 `VCardGeneratorUtil`이 VCARD 3.0 텍스트를 생성하며, 이미지가 있으면 base64 `PHOTO` 라인으로
  75자 폴딩하여 삽입.

> ⚠️ **이미지 다운로드 토큰은 사실상 중복(무의미)이다.** 아래 §4.6처럼 이미지는 `/uploads/**`로
> **공개** 서빙되므로, 같은 바이트를 5분 토큰(`image-download-url` / `downloads/image`)으로 보호해도
> 의미가 없다(토큰 없이도 공개 URL로 접근 가능). "이미지 공개"가 최종 결정이므로 이미지 토큰 경로는
> **논리적으로 중복**이며, vCard 다운로드 토큰만 실효성이 있다.
> 현재는 클라이언트 계약 유지를 위해 이미지 토큰 경로를 남겨두었고, 정리는 후속 과제로 둔다.

### 4.5 데이터 모델

- **`app_users`** (`UserEntity`): `id`(=`kakao_<카카오ID>`), `provider`, `email`, `nickname`, 생성/수정일.
- **`business_cards`** (`BusinessCardEntity`): 소유자 `user_id`, 이름/조직/연락처 등 명함 필드,
  `business_card_image_path`, `is_active`, `view_count`, vcf/이미지 다운로드 토큰 및 만료시각, 생성/수정일.
  - JSON 요청은 snake_case(`full_name` 등), 엔티티는 camelCase.
- **`stored_files`** (`StoredFileEntity`): 업로드 파일의 논리 경로(PK), 바이트(LONGBLOB), 콘텐츠 타입.
  `app.storage.type=db`일 때 이미지 저장소로 사용.

### 4.6 파일 저장 (2026-07-06 개편)

- 스토리지 추상화(`FileStorage`)로 구현 선택:
  - **`db` (기본/운영)**: `DbFileStorage` — 이미지 바이트를 Railway MySQL `stored_files`(LONGBLOB)에 저장.
    컨테이너 재배포에도 보존.
  - **`local` (개발)**: `LocalFileStorage` — 로컬 디스크 `./uploads/`.
  - 선택: `APP_STORAGE_TYPE` 환경변수.
- 파일명: `<cardId>_<timestamp>.<ext>`. `cardId`는 서버가 생성하거나 클라이언트가 지정할 수 있는데,
  **지정 시 반드시 UUID 형식이어야 한다**(`BusinessCardService.resolveCardId`가 검증). 따라서 파일명은
  UUID 기반이 보장되어 추측이 어렵다. (과거에는 임의 문자열 id를 허용해 이 전제가 약했다.)
- `/uploads/**` 는 `FileController`가 스토리지 경유로 서빙(정적 핸들러 대체) → 저장 백엔드 무관하게 동일 동작.

## 5. 모듈 ②: `dist_api_gateway` (리버스 프록시)

원래 커뮤니티/채팅 등 기능이 있던 서버였으나, "게이트웨이 최소화" 작업으로
**순수 프록시 역할만** 남긴 모듈. 기본 포트 `8080`.

### 5.1 동작

- `ApiGatewayController` 하나가 아래 경로를 businesscard_qr로 전달:
  `/api/**`, `/uploads/**`, `/swagger-ui.html`, `/swagger-ui/**`, `/v3/api-docs/**`
- `RestTemplate`으로 업스트림 호출. 요청/응답 본문을 **스트리밍**으로 전달(메모리 버퍼링 없음).
- **hop-by-hop 헤더 제거**, `X-Forwarded-Host/Proto/Port/For` 세팅.
- **multipart 파서 비활성화**(`spring.servlet.multipart.enabled: false`) — 멀티파트 본문을
  파싱하지 않고 원본 바이트 그대로 포워딩하기 위함 (배포 후 500 오류를 수정한 핵심 설정).
- 업스트림 대상 URL은 `APP_GATEWAY_BUSINESS_QR_URL`(구버전 `APP_GATEWAY_TARGET_BASE_URL`)로 지정.

### 5.2 게이트웨이의 역할 정리

- 클라이언트는 게이트웨이 URL만 바라봄 → 게이트웨이가 businesscard_qr로 중계.
- businesscard_qr가 `forward-headers-strategy: framework`로 X-Forwarded 헤더를 반영해,
  다운로드 URL 등 절대 URL이 게이트웨이 호스트 기준으로 생성됨.

## 6. 배포 (Railway)

1. `businesscard_qr` 서비스 생성 → `APP_MODULE=businesscard_qr`, DB/JWT/CORS 환경변수 설정.
2. `dist_api_gateway` 서비스 생성 → `APP_MODULE=dist_api_gateway`,
   `APP_GATEWAY_BUSINESS_QR_URL=<businesscard_qr URL>` 설정.
3. 클라이언트는 게이트웨이 URL을 base URL로 사용.

- 두 서비스 모두 동일한 루트 `Dockerfile`을 사용 (`start.sh`가 `APP_MODULE`에 따라 jar 선택 실행).
- `start.sh`는 Railway UI가 값에 따옴표를 붙이는 경우를 대비해 주요 환경변수의 감싸는 따옴표를 제거.
- 배포 안전장치: `RailwayDeploymentValidator`가 `RAILWAY_PROJECT_ID`가 있을 때
  - businesscard_qr: 기본 JWT 시크릿 사용을 차단
  - gateway: 업스트림 URL이 localhost면 차단

## 7. 요청 흐름 (전체 그림)

```
[Flutter 앱]
     │  Authorization: Bearer <JWT>
     ▼
[dist_api_gateway]  (/api/**, /uploads/**, swagger 프록시)
     │  RestTemplate 중계 (X-Forwarded-* 부여)
     ▼
[businesscard_qr]  (JWT 인증 → 명함 CRUD / 파일 저장 / vCard 생성)
     │
     ▼
 [MariaDB]  (이미지 바이트도 stored_files(LONGBLOB)에 저장 — 운영 기본 APP_STORAGE_TYPE=db)
     └ 개발 시 APP_STORAGE_TYPE=local 이면 [로컬 디스크 ./uploads]
```

## 8. 로컬 실행

```bash
./gradlew :businesscard_qr:bootRun     # 8081
./gradlew :dist_api_gateway:bootRun    # 8080
```

- 로컬 기본값: H2 인메모리 DB, 기본 JWT 시크릿, CORS는 localhost:3000/5173/8080 허용.
