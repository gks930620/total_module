실무에서 안 쓰는 이유
실무에서 저 방식 안 쓰는 이유는 프로젝트들이 서로 완전히 다른 서비스이기 때문이에요. 목적도 다르고, 팀도 다르고, 배포 시점도 달라서 한 서버에 합치는 게 오히려 이상한 상황이거든요.

SecurityFilterChain 여러 개 등록하면 돼요
Spring Security는 SecurityFilterChain을 여러 개 등록하고, URL 패턴별로 다른 체인을 타게 할 수 있어요.
java// projectA용 Security 설정
@Bean
@Order(1)
public SecurityFilterChain projectAFilterChain(HttpSecurity http) throws Exception {
    http
        .securityMatcher("/projectA/**")  // ← 이 URL만 이 설정 적용
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/projectA/api/public/**").permitAll()
            .anyRequest().authenticated()
        )
        .addFilterBefore(projectAJwtFilter, ...); // A 전용 필터
    return http.build();
}

// projectB용 Security 설정
@Bean
@Order(2)
public SecurityFilterChain projectBFilterChain(HttpSecurity http) throws Exception {
    http
        .securityMatcher("/projectB/**")  // ← 이 URL만 이 설정 적용
        .authorizeHttpRequests(auth -> auth
            .anyRequest().permitAll()  // B는 인증 없음
        );
    return http.build();
}
securityMatcher로 URL을 지정하면 각자 자기 URL에만 적용되고 서로 충돌 안 해요.

동작 방식
요청: /projectA/api/users
→ Order(1) 체인 securityMatcher 확인 → 매칭! → projectA 설정 적용

요청: /projectB/api/posts
→ Order(1) 체인 securityMatcher 확인 → 불일치
→ Order(2) 체인 securityMatcher 확인 → 매칭! → projectB 설정 적용
@Order 숫자가 낮을수록 먼저 확인해요.

공통으로 쓸 것 / 분리할 것
항목공통분리JWT 파싱 로직✅ 유틸 클래스로 공통화JwtFilter✅ 프로젝트별로SecurityFilterChain✅ 프로젝트별로UserDetailsService프로젝트별 유저 테이블이 다르면✅ 분리
JWT 토큰 파싱하는 로직 같은 건 공통 유틸로 빼고, 필터/체인 설정은 프로젝트별로 만드는 게 깔끔해요.

정리

securityMatcher + @Order 조합으로 프로젝트별 Security 설정을 완전히 독립적으로 가져갈 수 있어요. 충돌 걱정 안 해도 돼요!
