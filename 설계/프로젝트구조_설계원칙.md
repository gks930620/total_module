# 프로젝트 구조 설계원칙

> 2026-07-08 전면 개정. 방향의 근거는 [`아키텍처_방향결정.md`](아키텍처_방향결정.md) 참고 (충돌 시 그 문서가 우선).
> 다이어그램: [`프로젝트구조.svg`](프로젝트구조.svg)

## 핵심 한 줄

- **개발은 프로젝트별로 로컬에서 완성하고, 이 저장소에서는 "폴더 투하 + 등록"으로 배포한다.**
- 빌드는 하나(루트 Dockerfile → 이미지 1개), 실행은 모듈마다 하나(Railway 서비스 1개씩), 접근은 각자 공개 URL 직행.

## 1) 프로젝트 구조

```
/total_module (상위 폴더 — 실행 코드 없음, 스프링부트 아님)
  ├── .gitignore
  ├── Dockerfile             # 전체 조립 설계도 — 모든 모듈 jar를 이미지 하나에 담음
  ├── railway.toml           # Railway 빌드/헬스체크 설정
  ├── settings.gradle        # 하위 모듈 명단 (include 'businesscard_qr' 등)
  ├── build.gradle           # 하위 모듈 공통 규칙 (거의 비어있음)
  │
  ├── /businesscard_qr       # [모듈] 명함 API (Spring Boot — 독립 앱이면서 모듈)
  ├── /project_a             # [모듈] 향후 추가되는 프로젝트들...
  │
  ├── /businesscard_qr_app   # Flutter 앱 — Gradle 멤버 아님. 같이 두고 보기 위한 별개 프로젝트
  ├── /설계                   # 설계 문서 + 다이어그램 + 배포 가이드
  └── /클로드코드와응답        # 문제점/작업 기록 (Claude 산출물)
```

- 각 스프링부트 모듈은 **독립 실행 가능한 완전한 앱**이다 (자기 main 클래스, 자기 jar, 자체 gradlew).
- 게이트웨이/현관문 모듈은 **두지 않는다.** 각 모듈이 Railway에서 자기 공개 URL로 직접 트래픽을 받는다.

## 2) 실행/배포 모델

- Railway 프로젝트 1개 안에서 **모듈 1개 = 서비스 1개**. 같은 이미지를 쓰고 `APP_MODULE`로 역할 선택.
- jar을 하나로 합치지 않는다 — 합치면 프로젝트를 가져올 때마다 병합 수술(설정·빈·의존성 충돌)이 필요해져
  "완성본을 그대로 가져온다"는 컨셉이 깨진다.

## 3) 통합 프로젝트에 사이드 프로젝트를 추가할 때

1. 사이드 프로젝트는 **로컬에서 독립적으로 완성**한다 (각자 build.gradle, src 등)
2. 폴더를 total_module 루트에 넣고 `settings.gradle`에 `include` 추가
3. Dockerfile COPY/bootJar 대상 + start.sh APP_MODULE 목록에 추가
4. 배포 적합화 점검: env 주입(DB/시크릿), `PORT`, `/healthz`, 로컬 디스크 저장 금지
5. Railway에 서비스 추가 (`APP_MODULE=<모듈명>`) + Generate Domain
6. 클라이언트는 그 도메인을 직접 바라봄

(인텔리제이에서 모듈 인식: 루트 프로젝트에서 module 추가 → 스프링이면 Import Module from External Model(Gradle))

## 4) Git 작업 규칙

- 사이드 프로젝트마다 브랜치를 만들어 작업하고, PR로 리뷰 후 master에 머지한다.
- master push = Railway 재배포 트리거이므로, 배포에 필요한 env가 준비된 후 push한다.

## (역사) 2026-04-22 초기 설계 — 폐기됨

최초 설계는 "개발은 분리, **배포는 dist-server(게이트웨이) 하나로 합친다**"는 진짜 모놀리스였고,
`dist_api_gateway`가 모든 모듈을 품는 배포용 몸통이 될 예정이었다.
실제 구현이 "모듈별 Railway 서비스 분리"로 바뀌면서 이 개념은 폐기됐고,
게이트웨이는 존재 이유를 잃어 은퇴 수순([`아키텍처_방향결정.md`](아키텍처_방향결정.md) §4).
이 절은 "왜 게이트웨이가 있(었)는가"라는 질문에 답하기 위해 기록으로만 남긴다.
