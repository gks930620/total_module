## 🟡 1. 예외 핸들러가 표준 4xx를 500으로 덮음

**무엇**: `GlobalExceptionHandler`에 `@ExceptionHandler(Exception.class)`(→ 500)가 있어서, Spring이 원래 4xx로 처리하는 요청 오류까지 전부 500(서버오류)으로 나감.

**개념 — HTTP 상태코드**:
- **4xx = "클라이언트 잘못"** (예: 405 잘못된 메서드, 415 잘못된 Content-Type, 404 없는 경로, 400 잘못된 요청)
- **5xx = "서버 잘못"** (500 = 서버 내부 오류 = "우리 코드가 터졌다")
- Spring은 기본적으로 위 상황을 알아서 4xx로 응답한다. 그런데 `@ControllerAdvice`의 `@ExceptionHandler(Exception.class)`는 **모든 예외를 먼저 가로채서** Spring 기본 처리보다 우선한다 → 프레임워크가 던진 표준 예외까지 500으로 바꿔버림.

**왜 문제**:
- 클라이언트/앱이 "내가 요청을 잘못 보냈구나(4xx)"가 아니라 "서버가 터졌구나(5xx)"로 오해 → 잘못된 재시도.
- **Railway 모니터링에서 진짜 서버 장애와 "그냥 잘못된 요청"을 구분 못 함** (둘 다 500). 장애 알림이 노이즈로 뒤덮임.
- [`코드컨벤션.md`](../total_설계/코드컨벤션.md) §2 "표준 4xx를 500으로 뭉개지 않는다" 위반.

**시나리오**: 누가 `POST /api/community/123`(원래 GET인데 POST로)을 보냄 → 원래 405여야 하는데 500. 모니터링엔 "서버 오류 500" 뜸 → 진짜 장애인 줄 알고 확인하러 감.

**해결 방향**: `GlobalExceptionHandler`가 `ResponseEntityExceptionHandler`를 **상속**하면 프레임워크 표준 예외(405/415/404/400 등)의 상태코드를 보존한다. (catch-all Exception→500은 "정말 예상 못한 것"에만 남김)

## 🟡 2. 외부 API 호출이 트랜잭션(@Transactional) 안에 있음

**무엇**: OAuth 로그인 처리(`AppOAuth2Service.login`, `CustomOAuth2UserService.loadUser`)가 `@Transactional` 안에서 카카오/구글 서버로 HTTP 호출(토큰 검증)을 한다.

**개념 — 트랜잭션과 커넥션 풀**:
- `@Transactional` = "이 메서드 동안 DB 작업을 하나로 묶는다". 이 구간에서는 **DB 커넥션 1개를 잡고 있는다**.
- DB 커넥션은 **유한**하다. 이 앱은 HikariCP `maximum-pool-size: 10` → 동시에 10개까지만.
- 외부 HTTP 호출은 **느리고 불확실**(상대 서버 지연/타임아웃 가능).
- → 트랜잭션 안에서 HTTP를 하면, 그 느린 HTTP가 끝날 때까지 **DB 커넥션을 붙잡고 있게** 된다.

**왜 문제 (그리고 지금은 왜 안 터지나)**:
- 지금은 Hibernate가 **"첫 DB 쿼리 전까지는 커넥션을 안 잡는"**(지연 획득) 덕분에, HTTP가 트랜잭션 첫 줄이면 아직 커넥션을 안 쥐어서 최악은 피한다.
- **위험**: 나중에 누가 이 HTTP 호출 **앞에** DB 조회 한 줄을 추가하면, 그 순간부터 "커넥션 쥔 채 HTTP 대기"가 되어, 카카오 지연 시 **동시 로그인 10명이면 커넥션 풀 고갈 → 앱 전체 멈춤**.
- 트랜잭션 안 네트워크 I/O는 그 자체로 안티패턴(트랜잭션은 짧아야 함).

**시나리오**: (미래) 코드 수정으로 HTTP 앞에 사용자 조회가 생김 → 카카오 서버가 5초 지연 → 로그인 몰리면 10커넥션 다 점유 → 그동안 **다른 모든 요청(조회수·리뷰 등)도 DB 못 써서 대기**.

**해결 방향**: 외부 토큰 검증(HTTP)을 `@Transactional` **밖으로** 빼고, 검증 결과를 받은 뒤에만 트랜잭션(DB 저장)을 연다.

## 🟢 3. 리뷰 "하루 1회" 체크의 경쟁 조건 (TOCTOU)

**무엇**: 리뷰 작성 시 "오늘 이미 썼나?"를 `exists...()`로 확인 후 `save()`. 그 사이 틈으로 중복 생성 가능.

**개념 — TOCTOU (Time-Of-Check to Time-Of-Use)**:
- "확인하는 시점"과 "실제 쓰는 시점" 사이에 **틈**이 있으면, 그 틈에 다른 요청이 끼어들어 확인 결과가 무효가 되는 버그.
- 여기선: `exists(오늘 리뷰?) → false → save()`. 두 요청이 **거의 동시에** 오면, 둘 다 `exists=false`를 통과하고 둘 다 `save()` → **같은 날 리뷰 2개**.

**왜 문제**: "하루 1회" 비즈니스 규칙이 우회됨. DB에 **유니크 제약도 없어서** 막을 최후의 방어선도 없음.

**시나리오**: 사용자가 제출 버튼을 빠르게 두 번 누름 / 네트워크 느려서 앱이 자동 재시도 → 리뷰 2개 생성.

**해결 방향**:
- **DB 유니크 인덱스**(예: `user_id + doll_shop_id + 날짜`)를 걸면, DB가 원자적으로 두 번째 INSERT를 거부(중복키 오류) → 코드 틈과 무관하게 안전. (가장 확실)
- 또는 애플리케이션 락. 유니크 인덱스가 가장 단순·견고.

## 🟢 4. 조회수 증가가 원자적이지 않음 (Lost Update)

**무엇**: 게시글 상세 조회 시 `viewCount++` (자바 객체에서 값 읽고 +1 해서 저장). 동시 조회 시 일부 증가가 사라짐.

**개념 — Read-Modify-Write와 Lost Update**:
- `viewCount++`는 사실 3단계: **①현재값 읽기 → ②+1 → ③저장**.
- 두 요청이 동시에 ①에서 같은 값(예: 10)을 읽으면, 둘 다 11로 계산해서 저장 → **실제로는 12여야 하는데 11**. 한 번의 증가가 "유실(lost update)"됨.

**왜 문제**: 조회수가 부정확해짐. (다만 영향이 "숫자 살짝 틀림" 수준이라 우선순위 낮음)

**시나리오**: 인기 글을 여러 명이 동시에 열면 조회수가 실제보다 적게 카운트됨.

**해결 방향**: 자바에서 읽어서 더하지 말고, **DB에게 직접 원자적으로 시킨다**:
```sql
UPDATE community SET view_count = view_count + 1 WHERE id = ?
```
DB가 이 연산을 원자적으로 처리하므로 동시성에도 정확. (참고: businesscard의 조회수 증가는 이미 이 원자 UPDATE 방식)

**보충 — 실무 조회수는 보통 "dedup(중복 제거)"을 한다** (매 새로고침마다 +1 안 함)
- F5 연타·봇으로 뻥튀기되는 걸 막으려고, 실제 서비스는 조회를 **중복 제거**한다(예: 하루 1회, 세션 1회).
- 흔한 방법:
  - **쿠키/세션**: "이 글 봤음" 표시를 쿠키에 남기고 있으면 안 올림(하루/세션 단위). **비로그인도 됨** (네이버·티스토리 방식).
  - **조회 로그 테이블**: `(user_id 또는 ip, post_id, 날짜)` 유니크 테이블에 INSERT 성공했을 때만 +1. → **리뷰 "하루 1회"와 완전히 같은 dedup 패턴.**
  - **IP 기반**(비로그인) / **Redis TTL 키**(대규모).
- ⚠️ **"원자적 +1"과 "dedup"은 별개다**:
  - 원자 +1 = *올릴 때 정확히 올리기* (기술 이슈, 위 해결)
  - dedup = *무엇을 1회로 칠지* (기획 결정)
  - → dedup을 넣어도, 실제로 올리는 그 한 번은 **여전히 원자 +1**이어야 정확하다. 둘은 같이 간다.
- 이 프로젝트: 커뮤니티 상세는 **비로그인도 조회 가능** → user 기반 dedup만으론 익명 조회를 못 거른다 → **쿠키/IP 기반**이 현실적. 조회수가 핵심 기능은 아니니 **일단 원자 +1만** 하고, dedup은 원하면 나중에(쿠키 방식).

## 🟢 5. IllegalStateException을 전부 409로 매핑

**무엇**: `GlobalExceptionHandler`가 모든 `IllegalStateException` → 409(CONFLICT)로 응답.

**개념 — 예외 타입으로 상태코드 정하기**:
- 보통 "특정 예외 → 특정 HTTP 코드"로 매핑한다. 여기선 "이미 삭제된 리소스" 같은 상황을 409로 주려는 의도.
- 문제는 `IllegalStateException`이 **자바/스프링/라이브러리가 흔하게 던지는 아주 일반적인 예외**라는 것. 우리 의도와 **무관한** IllegalStateException(내부 버그, 설정 오류 등)까지 전부 409로 나감.

**왜 문제**: 실제로는 서버 내부 문제(500이어야 할 것)가 409("충돌")로 둔갑 → 원인 파악·모니터링 혼란.

**시나리오**: 내부 로직 버그로 `IllegalStateException`이 터졌는데 클라이언트는 409를 받아 "리소스 충돌인가?" 하고 엉뚱하게 대응.

**해결 방향**: "이미 삭제됨" 같은 **의도한 상황만을 위한 전용 커스텀 예외**(예: `AlreadyProcessedException`)를 만들어 그것만 409로 매핑. 일반 `IllegalStateException`은 500으로.

**🔗 1번(예외핸들러 4xx→500)과의 관계 — 사실 같은 병, 반대 방향**
- 둘 다 **"너무 넓은 예외 타입을 고정 상태코드로 매핑"** 하는 같은 안티패턴이다. 방향만 반대:
  - **1번**: 너무 넓은 `Exception.class` → 500 → 원래 **4xx**여야 할 게 **500**으로.
  - **5번**: 너무 넓은 `IllegalStateException` → 409 → 원래 **500**이어야 할 게 **409**로.
- **서로 충돌하진 않는다**: Spring은 던져진 예외에 **가장 구체적인** 핸들러를 고른다. `IllegalStateException`은 `Exception`보다 구체적이라 → 409로 감(5번), 그 외 프레임워크 예외는 특정 핸들러가 없어 → `Exception.class`(500)로 감(1번). 즉 같은 예외를 놓고 싸우진 않음.
- 하지만 **함께 보면** 이 핸들러가 예외를 **양방향으로 잘못 분류**하고 있는 것: 4xx가 500으로도, 500이 409로도 샌다.
- **공통 해결 철학(같이 정리하면 깔끔)**: 넓은 타입(Exception/IllegalStateException)에 기대지 말고 **의도한 전용 예외를 구체적으로** 매핑한다.
  - 1번 → `ResponseEntityExceptionHandler` 상속으로 프레임워크 표준 예외의 4xx 보존
  - 5번 → 전용 예외(`AlreadyProcessedException` 등)만 409, 일반 `IllegalStateException`은 500
  - → 두 개를 함께 고치면 "예외 → 상태코드" 매핑이 의도대로 정리된다.

## 프론트 A. 토큰 refresh 실패 시 대기 요청이 영영 안 끝남 (hang)

**무엇**: 토큰이 만료돼 재발급(refresh) 중일 때 들어온 다른 요청들은 "대기 큐"에 들어가는데, **refresh가 실패하면** 그 큐를 콜백 호출 없이 비워버려서 대기 중이던 요청 Promise가 영원히 안 끝남.

**개념 — Promise는 반드시 결말이 나야 한다**:
- JS의 `Promise`(비동기 작업)는 언젠가 **resolve(성공) 또는 reject(실패)** 로 끝나야 한다. 둘 다 안 되면 그걸 `await`하던 코드는 **영원히 멈춰(pending)** 있다.
- 여기선 refresh가 진행 중이면 다른 요청들이 "refresh 끝나면 실행" 콜백을 큐에 넣고 대기. refresh **성공** 시엔 콜백들을 실행해서 정상 종료되는데, **실패** 시엔 큐를 그냥 비워(콜백 호출 X) → 그 Promise들이 미결로 남음.

**왜 문제**: 그 요청을 기다리던 화면이 **스피너/`submitting` 상태에서 안 풀림**(무한 로딩처럼 보임).

**시나리오**: 토큰 만료 + 사용자가 리뷰 제출(이미지 업로드 등 동시 인증요청 여러 개) + 마침 refresh도 실패(리프레시 토큰도 만료) → 일부 요청이 영영 안 끝나 제출 버튼이 계속 "처리 중".

**해결 방향**: refresh **실패** 시에도 대기 콜백들을 **실패로 처리**(각 Promise를 reject 또는 원래 401 응답으로 resolve)해서 반드시 결말을 내준다.

## 프론트 B. 콜드스타트 안내가 없음 (App Sleeping과 직접 관련)

**무엇**: fetch에 타임아웃도, "서버 깨우는 중" 안내도 없음. App Sleeping을 켰으니 첫 요청이 10~30초 걸리는데, 그동안 화면이 아무 피드백 없이 멈춘 것처럼 보임.

**개념 — App Sleeping + 콜드스타트**:
- App Sleeping으로 앱이 잠들면, **첫 요청이 앱을 깨우느라 10~30초** 걸린다(정상). Railway가 그 요청을 큐에 잡아뒀다가 깨어나면 처리하므로 **결국 응답은 온다**(무한행 아님).
- 문제는 그 10~30초 동안 사용자에겐 그냥 "..."/"불러오는 중"만 보인다 → 멈춘 줄 알고 이탈.

**왜 문제**: 정상 동작인데 **사용자에겐 고장처럼** 보임. (특히 앱이 마운트되며 부르는 `/api/my/info`가 이 첫 요청이 되기 쉬움)

**시나리오**: 한동안 아무도 안 쓴 뒤 첫 방문자 → 로그인 확인 요청이 25초 pending → 헤더·화면이 멈춘 듯 보임 → "고장났네" 하고 나감.

**해결 방향**:
- 로딩이 일정 시간(예: 3~5초) 넘으면 **"서버를 깨우는 중이에요, 잠시만요"** 안내 표시.
- (선택) fetch 타임아웃 + 5xx/네트워크 오류 시 자동 1회 재시도(콜드스타트 대비).
