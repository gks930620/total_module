# ERP/CMS 공통기능 및 보일러플레이트 적용 계획

## 1) 목표
- ERP와 CMS를 각각 만들더라도 **공통 기반(인증, 권한, 코드, 로그, 파일, 배포 설정)** 은 하나로 유지한다.
- 현재 `spring_csr_boiler_component` 구조를 최대한 재사용해, 신규 기능을 도메인별로 확장한다.

---

## 2) ERP 공통기능 (권장 기본 세트)

### 필수
- 인증/인가: 로그인, 역할(Role), 권한(Permission), 메뉴별 접근제어
- 조직/사용자 관리: 회사, 부서, 사용자, 직책, 사용자 상태(활성/잠금)
- 공통코드 관리: 코드그룹/코드(예: 주문상태, 결재상태, 거래처유형)
- 기준정보(Master) 관리: 거래처, 품목, 창고, 계정과목 등 도메인 공통 마스터
- 결재/승인 워크플로우: 기안/승인/반려, 승인선 템플릿
- 감사 로그(Audit): 누가/언제/무엇을 변경했는지 저장
- 알림: 결재 요청, 반려, 상태변경 알림 (앱/웹)
- 엑셀 업/다운로드: 대량 등록/정산용
- 배치/스케줄러: 집계, 마감, 비동기 후처리

### 있으면 좋은 기능
- 다국어/다통화, 회계기간, 멀티법인
- 통합 인터페이스(외부 API, EDI, 메일)
- 캐시/성능 모듈(핫 데이터)

---

## 3) CMS 공통기능 (권장 기본 세트)

### 필수
- 콘텐츠 모델링: 게시글/페이지/배너/팝업/카테고리
- 에디터 + 미디어 라이브러리: 이미지/파일 업로드, 썸네일, 첨부
- 콘텐츠 상태 관리: 임시저장/발행/예약발행/종료
- 버전관리: 변경 이력, 롤백
- SEO 기본: slug, meta title/description, OG
- 권한/워크플로우: 작성자/검토자/발행자 분리
- 메뉴/네비게이션 관리
- 템플릿/레이아웃 관리

### 있으면 좋은 기능
- A/B 테스트, 개인화 노출
- 통계 대시보드(조회수, 유입)
- 다국어 콘텐츠

## 5) 현재 보일러플레이트 재사용 포인트

이미 확보된 공통 기반:
- 인증/보안: `jwt/*`, `SecurityConfig`, OAuth2, Refresh Token
- 응답/예외 표준화: `common/dto/*`, `common/exception/*`
- 파일관리: `file/*` + 저장 전략(로컬/Supabase)
- 실시간 통신: `stomp/*`
- CSR 프론트 기본기: `frontend/src/context/AuthContext.jsx`, `frontend/src/lib/http.js`

즉, 신규 구축은 “기반 재작성”이 아니라 **도메인 모듈 추가** 중심으로 진행 가능하다.

## 6) 보일러플레이트 적용 계획 (단계별)

### Phase 1. 공통 코어 고도화 (ERP/CMS 공용)
1. 감사/Audit 기반 추가
   - `common/audit/` 신설
   - `createdBy`, `updatedBy`, 변경이력 테이블(예: `audit_log`) 추가
2. 권한 모델 세분화
   - 현재 `UserEntity.roles(List<String>)`를 유지하되,
   - 점진적으로 `Role`, `Permission`, `RolePermission`, `UserRole` 테이블 분리
3. 공통코드 모듈 추가
   - `common/code/` (코드그룹/코드 CRUD + 캐시)

### Phase 2. CMS 공통 모듈 먼저 구축
1. `cms/content` 모듈
   - 콘텐츠 엔티티(타입, 상태, 발행시간, slug)
2. `cms/category`, `cms/menu`
3. `cms/workflow`
   - 검토/발행 승인 플로우
4. 파일 모듈 연동
   - 기존 `file.RefType`에 `CMS_CONTENT` 등 타입 확장

### Phase 3. ERP 공통 모듈 구축
1. `erp/master`
   - 거래처/품목/창고 등 기준정보
2. `erp/org`
   - 회사/부서/직책
3. `erp/approval`
   - 결재선/결재문서/이력
4. `erp/notification`
   - 결재/상태 변경 알림 (기존 STOMP 재사용 가능)

### Phase 4. 프론트 통합
1. 라우트 영역 분리
   - `/admin/*`, `/cms/*`, `/erp/*`
2. 권한 기반 메뉴 렌더링
   - 로그인 후 권한 API 응답으로 사이드 메뉴 구성
3. 공통 컴포넌트화
   - 목록/검색/페이징/폼 검증/파일업로드 컴포넌트 공유

---

## 8) 우선순위 제안 (현실적인 시작점)
1. **권한 + 공통코드 + Audit** 먼저
2. 이후 **CMS 콘텐츠 관리**를 첫 업무 모듈로 구현
3. ERP는 **조직/결재/기준정보** 순으로 확장

이 순서가 좋은 이유:
- 공통기반이 먼저 안정화되어, 이후 ERP/CMS 도메인 구현 속도가 빨라짐
- CMS가 상대적으로 트랜잭션 복잡도가 낮아 보일러 검증에 유리함

---

## 9) 완료 기준(Definition of Done)
- 공통 API 응답/에러 포맷이 신규 모듈에도 100% 동일
- 권한 없는 요청이 화면/백엔드 모두에서 차단
- 변경 이력(Audit) 조회 가능
- 파일 업로드/연결/삭제 정책이 CMS/ERP 양쪽에서 동일
- 운영 프로필(`application-prod.yml`) 기준으로 배포/헬스체크/로그 확인 완료
