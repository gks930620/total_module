## 2. 신규 지적 사항 (문서 미반영 / 문서-코드 불일치)

### 🟡 M1. 클라이언트가 명함 `id`(PK)를 직접 지정 가능 → 공개 이미지의 "파일명 추측 어려움" 전제 약화 + PK 충돌 시 500
- **위치**: `card/dto/BusinessCardUpsertRequest.java:15`(`private String id;` 검증 없음),
  `BusinessCardService.createBusinessCard:65-67`(요청 id가 있으면 그대로 cardId 사용),
  `storeImage:244`(파일명 = `<cardId>_<timestamp>.ext`)
- **내용**:
  1. `프로젝트_개요.md §4.5`와 `문제점_정리 #3`는 `/uploads/**` 공개를 유지하는 근거로
     **"파일명이 UUID+timestamp라 추측이 어렵다"**를 든다. 그러나 cardId는 UUID가 아니라
     **클라이언트가 임의 지정**할 수 있어(예: `id:"1"`), 이미지 경로가 `.../1_<timestamp>.jpg`로
     상당히 예측 가능해진다. → **문서의 안전성 근거가 코드와 어긋남.**
  2. 다른 사용자가 이미 쓴 id로 생성 시 PK 충돌 → `DataIntegrityViolationException`이
     전용 핸들러 없이 일반 `Exception` 핸들러로 떨어져 **500**(413처럼 매핑 안 됨). 존재하는 id를
     500 여부로 탐지할 수 있는 소소한 정보 노출도 있음.
- **권장**: 생성 시 서버가 항상 UUID를 강제(요청 id 무시)하거나, 최소한 문서의 "추측 어려움" 근거를
  정정. PK 충돌은 전용 예외 → 409로 매핑.

### 🟡 M2. 게이트웨이 루트 `/`가 내부 업스트림 URL을 인증 없이 노출
- **위치**: `gateway/controller/GatewayStatusController.root():15-22`
- **내용**: 게이트웨이 모듈은 Spring Security 의존성이 없어 `/`가 완전 공개인데,
  응답에 `targetBaseUrl`(businesscard_qr **내부 서비스 URL**)을 그대로 담아 반환한다.
  내부 토폴로지/주소 노출이며 공격 표면 정보가 된다.
- **권장**: 루트 응답에서 `targetBaseUrl` 제거(상태만 반환). 헬스체크는 `/healthz`로 충분.

### ⚪ L1. Supabase 참조 주석 잔존 (문서-코드 불일치)
- **위치**: `card/storage/FileStorage.java:8`, `card/controller/FileController.java:15-16`,
  `common/config/WebConfig.java:17`
- **내용**: `진행사항_2026-07-06.md §1`은 "외부 스토리지(Supabase)가 **아니라** Railway MySQL로
  완성"이라 명시했는데, 위 3곳 javadoc/주석은 여전히 저장 백엔드를 **"Supabase"**로 서술한다.
  실제 구현은 `DbFileStorage`(DB) / `LocalFileStorage`(디스크)뿐 — Supabase 연동 없음. 오해 유발.
- **권장**: 주석의 "Supabase" → "DB(MySQL)/로컬"로 정정.

### ⚪ L2. view-count가 여전히 인증 필요 → 설계 취지("보는 사람이 올림")와 부분 불일치
- **위치**: `SecurityConfig`(`/api/**` authenticated), `BusinessCardController.incrementViewCount:80`
- **내용**: #9로 **소유자 제한**은 제거됐으나, 엔드포인트는 여전히 `/api/**`(인증 필요)라
  **로그인 사용자만** 조회수를 올릴 수 있다. QR을 스캔한 **비로그인 방문자**는 카운트 불가 →
  `문제점_정리 #9`의 근본 취지("조회수는 보는 사람이 올리는 값")는 여전히 미충족.
- **권장**: 의도된 동작이면 문서에 "인증 사용자만 집계"임을 명시. 아니면 공개/토큰 경로로 재설계.

### ⚪ L3. 예외 핸들러 한글 문자열 매칭 — 사실상 죽은 경로 (제거 후보)
- **위치**: `GlobalExceptionHandler.determineStatusFromMessage/determineErrorCodeFromMessage`
- **내용**: #11로 "유지" 처리됐으나, 현재 서비스 계층은 대부분 `BusinessRuleException`/
  `EntityNotFoundException` 등 커스텀 예외를 던진다(`BusinessException` 핸들러가 상태코드 결정).
  즉 `IllegalArgumentException` 한글 매칭 분기는 실제로 타는 곳이 거의 없는 레거시. "후속 과제"로만
  둘 게 아니라 **제거 가능 여부를 적극 검토**할 만하다(문자열 매칭은 문구 변경 시 조용히 오동작).

### ⚪ L4. CORS `allowCredentials(true)` + `Set-Cookie` 노출이 JWT(무상태) 방식과 불필요하게 중복
- **위치**: `common/config/WebConfig.java:31-32`
- **내용**: 인증이 `Authorization: Bearer` 무상태 JWT라 쿠키를 쓰지 않는데도 `allowCredentials(true)`
  및 `Set-Cookie` 노출을 설정. 기능 문제는 아니나 불필요하고, credentials=true는 향후 오리진 설정
  실수 시 위험을 키운다. 정리 권장.

### ⚪ L5. 문서 위치 정리 과도기 흔적 (중복 가능성)
