## 🟡 중간 — 설계 의도와 실제 동작 불일치

### 1. 조회수 증가 API가 여전히 인증을 요구함 → #9 수정의 의도 미달성

- **위치**: `common/config/SecurityConfig.java:37` (`/api/**` → `authenticated()`),
  `card/controller/BusinessCardController.java:80` (`POST /{cardId}/view-count`),
  `BusinessCardService.incrementViewCount:104`
- **내용**: `진행사항_2026-07-06.md` #9는 "조회수는 **보는 사람**이 올리는 값"이라는 근거로
  소유자 제한을 제거했다고 밝힘. 실제로 서비스 계층에서 `getOwnedActiveCard` → `getActiveCard`로
  바뀐 것은 맞음. **그러나** 이 엔드포인트는 여전히 `/api/**` 규칙에 걸려 **JWT 인증이 필수**임.
  명함을 QR로 스캔하는 실제 방문자는 이 앱의 로그인 사용자가 아닐 가능성이 크므로,
  "보는 사람이 조회수를 올린다"는 시나리오는 **실현되지 않음**. 결국 소유자 제한만 풀렸을 뿐,
  로그인한 사용자들끼리만 조회수를 올릴 수 있음.
- **판단**: 수정의 목적과 결과가 어긋남. 둘 중 하나를 택해야 함.
  - (a) 의도대로 "방문자 집계"라면 → 조회수 엔드포인트를 공개 경로(또는 다운로드처럼 토큰 경로)로 이동.
  - (b) "로그인 사용자만 집계"가 최종 의도라면 → 문서의 근거 서술을 실제 동작에 맞게 정정.

### 2. 리프레시 토큰 "회전(rotation)"은 실제 회전이 아님

- **위치**: `auth/service/AuthService.java:49` (`refresh`), `common/security/JwtTokenProvider.java`
- **내용**: 문서는 `POST /api/auth/refresh`를 "재발급(**회전**)"으로 기술. 코드는 유효한 refresh 토큰이면
  새 access/refresh 토큰을 발급함. 그러나 리프레시 토큰이 **무상태 JWT**이고 서버에 저장/블랙리스트가
  전혀 없으므로, **기존 리프레시 토큰이 폐기되지 않고 만료(기본 14일)까지 그대로 유효**함.
  즉 "회전"의 핵심(구 토큰 무효화)이 성립하지 않음. 재발급은 맞지만 회전은 아님.
- **영향(보안)**: 리프레시 토큰이 유출되면 14일간 임의 재발급이 가능하며 **서버가 이를 취소할 수단이 없음**.
  access 토큰을 1시간으로 짧게 둔 이점이 반감됨.
- **판단**: 현재 규모에선 수용 가능하나, 문서에서 "회전"이라는 단어는 오해를 부름.
  진짜 회전이 필요하면 refresh 토큰 저장소(jti 화이트리스트/버전 컬럼) 도입 검토, 아니면 용어를 "재발급"으로 정정.

### 3. `/uploads/**` 공개 서빙이 이미지 다운로드 토큰 설계를 완전히 무력화

- **위치**: `card/controller/FileController.java:24` (`/uploads/**` 공개),
  `SecurityConfig.java:38` (`anyRequest().permitAll()`),
  `BusinessCardService.toResponse:189` (`imageUrl`을 공개 URL로 반환)
- **내용**: `문제점_정리.md` #3에서 이미 인지·의도 유지로 분류됨. 다만 검수자2 관점에서 재차 지적:
  명함 응답(`imageUrl`)이 `/uploads/business-card-images/<파일명>`을 **인증·토큰 없이** 노출하고
  `FileController`가 그대로 서빙함. 반면 `image-download-url` + 5분 토큰(`/downloads/image?token=`)
  메커니즘은 **동일한 바이트**를 보호하려 함. 두 경로가 공존하는 한 토큰 경로는 **보안적으로 무의미**하며
  순수 중복 로직임(엔티티에 토큰 컬럼 2쌍 + 발급 API 2개 유지비용).
- **판단**: "이미지는 공개"가 최종 결정이면 → `image-download-url`/`downloads/image`/`imageDownloadToken`
  일체를 제거해 설계를 단순화(다운로드는 vCard만 토큰). "보호 대상"이면 → `/uploads/**` 정적 공개를 막고
  토큰 경로로만 제공. **어느 쪽이든 지금처럼 절반씩 걸쳐 있으면 안 됨.**

## ⚪ 낮음 — 문서·주석 정리 / 일관성

### 4. 코드 주석·Javadoc에 실재하지 않는 Supabase 잔재

- **위치**:
  - `card/storage/FileStorage.java:8` — "구현체(로컬 디스크 / **Supabase Storage**)"
  - `card/storage/LocalFileStorage.java:15` — "`supabase.enabled` 가 true가 아닐 때 사용" + "**기본 구현**"
  - `card/controller/FileController.java:15` — "저장 백엔드(로컬/ **Supabase**)를 통일"
  - `common/config/WebConfig.java:17` — "로컬/**Supabase** 통일"
- **내용**: 실제 구현은 `DbFileStorage`(기본, `@ConditionalOnMissingBean`) / `LocalFileStorage`이며
  선택 키는 `app.storage.type`임. `supabase.enabled` 프로퍼티나 Supabase 구현은 **코드에 존재하지 않음**.
  특히 `LocalFileStorage` Javadoc의 "**기본 구현**"은 사실과 반대(기본은 `DbFileStorage`)라 오독 위험.
- **판단**: 주석을 실제 구조(DB/로컬, `app.storage.type`)에 맞게 정정. #1(이미지 유실)과 얽혀
  "Supabase 연동돼 있겠지" 착각을 유발할 수 있어 우선순위 있게 정리 권장.

### 5. 설계 문서 내부 불일치 — 요청 흐름도의 저장소 표기

- **위치**: `설계/프로젝트_개요.md` 7절 요청 흐름도 마지막 줄 `[MariaDB] + [로컬 디스크 ./uploads]`
- **내용**: 같은 문서 4.6절과 `진행사항_2026-07-06.md`는 **기본 저장소가 DB(`stored_files`)**라고 명시하는데,
  흐름도만 "로컬 디스크 ./uploads"로 남아 있음. 문서 내에서 상호 모순.
- **판단**: 흐름도를 DB 저장(운영 기본)으로 갱신, 로컬 디스크는 개발용(`APP_STORAGE_TYPE=local`)으로 병기.

### 6. `existsById` 유지 근거의 과장

- **위치**: `common/security/JwtAuthenticationFilter.java:52`, 근거 서술: `진행사항_2026-07-06.md` 7항
- **내용**: "삭제/**차단**된 사용자의 토큰을 즉시 무효화하기 위한 보안 의도"라 기술. 그러나 사용자 엔티티에
  **차단(block) 상태 필드나 블랙리스트가 없음**(`UserEntity`). 실제로 무효화되는 경우는 사용자 **행 삭제**뿐.
  즉 근거의 "차단"은 현재 구현되지 않은 기능을 전제로 함.
- **판단**: 서술을 "삭제된 사용자 즉시 반영"으로 축소하거나, 차단 기능이 로드맵이면 그 사실을 명시.

### 7. 스트리밍 개선이 게이트웨이에서 멈춤(원본 서빙은 전량 메모리)

- **위치**: `card/controller/FileController.java` / `card/storage/DbFileStorage.java:31` (`byte[]` 전량 로드),
  `BusinessCardService.downloadImage:183` (`byte[]` 반환)
- **내용**: #4로 게이트웨이는 `byte[]` 버퍼링 → 스트리밍으로 개선됨. 그러나 실제 파일을 생산하는
  businesscard_qr 측 `FileController`/`downloadImage`는 여전히 이미지 전체를 힙에 적재해 반환함.
  명함 이미지(≤10MB)라 당장 문제는 없으나, "메모리 효율" 개선이 게이트웨이 한쪽에만 적용된 셈.
- **판단**: 현행 규모에선 수용 가능. 향후 큰 파일이 들어오면 origin도 `InputStreamResource`/스트리밍 고려.
  DB 스토리지 특성상 완전 스트리밍은 어려우므로 "이미지 크기 상한(현재 10MB)"을 설계상 의존 조건으로 명시 권장.

### 8. "BUILD SUCCESSFUL" 주장 — 이번 검수에서 재검증하지 않음

- **내용**: `진행사항_2026-07-06.md` 0절은 `./gradlew build` 성공(통합 12 + JWT 3 + 게이트웨이 3)을 주장.
  이번 검수는 정적 리뷰로 진행했고 **빌드/테스트를 재실행하지 않음**(환경상 `JAVA_HOME` 수동 설정 필요).
- **판단**: 병합 전 `JAVA_HOME`을 Corretto 17로 지정 후 `./gradlew build` 1회 재확인 권장.
  코드상 컴파일/컨텍스트 로딩을 막을 결함은 발견되지 않음
  (게이트웨이 `@EnableConfigurationProperties(GatewayRoutingProperties.class)` 확인, 스토리지 빈 조건부 등록 확인).

## 참고: 코드-설계 정합성이 잘 맞는 부분 (그대로 유지)

- 스토리지 추상화(`FileStorage`) + `@ConditionalOnMissingBean` 기본 DB 선택 — 설계 의도와 일치.
- 다운로드 토큰(5분 만료) 검증 로직(`isVcfTokenValid`/`isImageTokenValid`)이 만료·일치 모두 확인.
- 게이트웨이 hop-by-hop 헤더 제거 + `X-Forwarded-*` 세팅 + `Content-Length` 제거 후 스트리밍 — 문서와 일치.
- `RailwayDeploymentValidator`(양 모듈) 배포 안전장치 — 문서 6절과 일치.
- 업로드 콘텐츠 타입 화이트리스트(`image/*` + 확장자) 및 413 핸들러 — #14와 일치.
- `Content-Disposition`을 UTF-8로 인코딩(`filename(..., UTF_8)`)해 한글 파일명 안전.
- `LocalFileStorage.resolvePhysical`의 경로 탈출(`../`) 방지 검증 존재.
