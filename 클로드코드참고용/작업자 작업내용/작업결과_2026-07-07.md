### ✅ A. 조회수 증가 API 공개 경로화 (검수1 L2 / 검수2 #1)

  - `SecurityConfig`: `HttpMethod.POST, /api/business-cards/*/view-count` → `permitAll()` 추가(`/api/**` authenticated 앞).

## 2. 검수자 주장 재검증 — "PK 충돌 → 500"은 부정확했음 (검수1 M1)

- **검수자1 주장**: 클라이언트가 명함 `id`(PK)를 임의 지정 가능(`id:"1"`)하며,
  기존 id 재사용 시 **PK 충돌 → `DataIntegrityViolationException` → 500**.
- **실제 코드 재검증 결과(검수자 주장과 다름)**:
  - `BusinessCardEntity`는 `@Id String id`(assigned id, `@Version` 없음, `Persistable` 미구현).
  - 따라서 Spring Data JPA `save()`는 `isNew()==false`로 판단해 **`persist`가 아니라 `merge`를 호출**한다.
  - `merge`는 해당 id가 이미 존재하면 500을 던지지 않고 **UPDATE로 기존 행을 덮어쓴다**.
    → 즉 다른 사용자의 명함 id를 지정하면 **소유자(`user_id`)를 포함해 그 카드를 조용히 덮어쓰는
    소유권 탈취(IDOR)** 가 가능. 검수자가 지적한 500보다 **더 심각**하다.
- **처리(오너 결정: UUID 검증 + 409, 실제 위험에 맞춰 강화)**:
  - `BusinessCardService.resolveCardId(requestedId)` 신설:
    - 미지정 → 서버가 `UUID.randomUUID()` 생성.
    - 지정 → `UUID.fromString`으로 **UUID 형식 검증**(형식 오류 시 `BusinessRuleException` → 400).
      오프라인 우선 클라이언트가 미리 생성한 UUID는 그대로 허용.
    - **`existsById`로 존재 여부 확인** → 충돌 시 `DuplicateResourceException`(409). **덮어쓰기 원천 차단.**
  - `GlobalExceptionHandler`: `DataIntegrityViolationException` → **409** 핸들러 추가
    (동시 생성 레이스 등 잔여 경로의 백스톱. 기존엔 일반 `Exception`→500으로 떨어졌음).
  - 통합 테스트 2건 추가:
    - `createBusinessCard_rejectsNonUuidId`(`id:"1"` → 400)
    - `createBusinessCard_rejectsExistingId_withoutOverwrite`(타 사용자 id 지정 → 409 + 원본 카드 불변 검증)
  - `설계/프로젝트_개요.md §4.5`의 "파일명 UUID라 추측 어려움" 근거를 실제 동작(UUID 강제)으로 정정.

## 3. 문서-코드 정합성 정리 (검수1 L1 / 검수2 #4·#5·#6·#2)

### ✅ D. Supabase 잔재 주석 제거 (검수1 L1 / 검수2 #4)

실제 구현은 `DbFileStorage`(운영 기본, `app.storage.type=db`) / `LocalFileStorage`(개발)뿐이고
Supabase 연동/`supabase.enabled` 프로퍼티는 코드에 **존재하지 않음**. 아래 주석/Javadoc 정정:

- `card/storage/FileStorage.java` — "로컬 디스크 / Supabase Storage" → "DB(MySQL) / 로컬 디스크".
- `card/storage/LocalFileStorage.java` — "**기본 구현**"(사실은 DB가 기본) 및 "`supabase.enabled`" 서술 삭제 →
  "개발용, `app.storage.type=local` 일 때 사용, 운영 기본은 `DbFileStorage`".
- `card/controller/FileController.java` — "로컬/Supabase" → "DB/로컬".
- `common/config/WebConfig.java` — "로컬/Supabase 통일" → "DB/로컬 통일".
- 재검증: `grep -i supabase businesscard_qr/src` → **0건**.

### ✅ E. 설계 문서 내부 정정 (검수2 #5·#6·#2)

- `프로젝트_개요.md §7 요청 흐름도`: `[MariaDB] + [로컬 디스크 ./uploads]` →
  **DB 저장(운영 기본)** 표기 + 개발 시 로컬 디스크 병기(문서 내 §4.6과의 모순 해소).
- `진행사항_2026-07-06.md §7 existsById 근거`: "삭제/**차단**된 사용자" → "**삭제된** 사용자"로 축소
  (`UserEntity`에 차단 상태 필드 없음. 차단은 로드맵으로 명시).
- **리프레시 토큰 "회전(rotation)" 용어 정정**(검수2 #2): 무상태 JWT라 구 토큰 폐기가 불가하므로

  정정 및 경고 추가(진짜 회전이 필요하면 jti 화이트리스트/버전 컬럼 도입 필요).
