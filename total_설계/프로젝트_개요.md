## 1. 한 줄 요약

카카오 로그인으로 인증한 사용자가 **디지털 명함(business card)을 등록/조회/수정/삭제**하고,
명함을 **vCard(.vcf) 파일 또는 이미지로 다운로드**할 수 있게 해주는 백엔드 서비스 + Flutter 앱.

total_module 저장소 자체는 "로컬에서 완성한 프로젝트들을 모아 하나의 Railway 프로젝트로 배포"하는
통합 저장소이며, 명함QR은 그 첫 번째 입주 프로젝트다.

---

## 2. 구조 (요청 흐름)

```
[Flutter 앱 (businesscard_qr_app)]
     │  HTTPS + Authorization: Bearer <JWT>
     │  base URL = https://businesscardqr-production.up.railway.app  (직접 연결)
     ▼
[businesscard_qr (Spring Boot, Railway 서비스)]
     │  인증(카카오→JWT) · 명함 CRUD · 파일 저장 · vCard 생성
     ▼
[Railway MySQL]  (이미지 바이트도 stored_files(LONGBLOB)에 저장 — APP_STORAGE_TYPE=db)
```

- 게이트웨이 없음. 절대 URL(QR/이미지/다운로드)은 `APP_PUBLIC_BASE_URL` env 기준으로 생성.
- 저장소 구조·배포 모델은 [`프로젝트구조_설계원칙.md`](프로젝트구조_설계원칙.md), 다이어그램은
  [`서버아키텍처.svg`](서버아키텍처.svg) / [`프로젝트구조.svg`](프로젝트구조.svg).

## 4. businesscard_qr 모듈

명함 도메인의 비즈니스 로직 담당. 기본 포트 `8081`(로컬), Railway에서는 `PORT` 주입.

### 4.1 패키지 구조 (도메인별)

```
com.businesscard
├── auth/      카카오 로그인 → JWT 발급/재발급
├── user/      사용자 동기화/저장 (app_users)
├── card/      명함 CRUD, 이미지 저장(storage/), vCard/이미지 다운로드, PublicUrlResolver
└── common/    보안(JWT 필터), 예외 처리, CORS, 배포 검증기(RailwayDeploymentValidator)
```

### 4.2 인증 흐름

1. 앱이 카카오 SDK로 카카오 액세스 토큰 획득 → `POST /api/auth/kakao`
2. 서버가 카카오 `/v2/user/me`로 검증(+선택: `APP_KAKAO_EXPECTED_APP_ID`로 app_id 검증) → 사용자 upsert(`kakao_<id>`)
3. 자체 JWT **액세스(기본 1시간) + 리프레시(기본 14일)** 발급
4. 이후 요청은 `Authorization: Bearer <jwt>`. 만료 시 `POST /api/auth/refresh`
   (무상태 재발급 — 구 리프레시 토큰의 서버측 무효화는 없음)
5. 앱은 401 수신 시 refresh → 실패 시 카카오 재로그인으로 자동 복구

### 4.3 주요 API

| 메서드 | 경로 | 인증 | 설명 |
|--------|------|------|------|
| POST | `/api/auth/kakao` | 공개 | 카카오 로그인 → JWT 발급 |
| POST | `/api/auth/refresh` | 공개 | 리프레시 토큰으로 재발급 |
| POST | `/api/users/sync` | 필요 | 사용자 프로필 동기화 |
| GET/POST | `/api/business-cards` | 필요 | 목록 / 생성(multipart: `payload`+`businessCardImage`) |
| GET/PUT/DELETE | `/api/business-cards/{id}` | 필요 | 단건 조회 / 수정 / 소프트 삭제 |
| POST | `/api/business-cards/{id}/view-count` | 공개 | 조회수 증가 (원자적 UPDATE, 비로그인 방문자 포함) |
| GET | `/api/business-cards/{id}/vcf-download-url`·`/image-download-url` | 필요 | 5분 토큰 다운로드 URL 발급 |
| GET | `/api/business-cards/{id}/downloads/vcf`·`/image?token=` | 공개 | 토큰 검증 후 다운로드 |
| GET | `/uploads/**` | 공개 | 이미지 서빙 (FileController, 스토리지 경유) |
| GET | `/healthz`, `/swagger-ui.html` | 공개 | 헬스체크 / API 문서 |

- 절대 URL은 `PublicUrlResolver`가 `APP_PUBLIC_BASE_URL` 기준으로 생성 (미설정 시 X-Forwarded 폴백 + WARN).
- 이미지 다운로드 토큰은 `/uploads/**` 공개와 중복(실효성 없음) — 앱 계약 유지를 위해 존치, 후속 정리 과제.

### 4.4 데이터 모델

- **`app_users`**: `id`(=`kakao_<카카오ID>`), provider, email, nickname, 생성/수정일
- **`business_cards`**: 소유자, 명함 필드(snake_case JSON 계약), `is_active`(소프트 삭제), `view_count`,
  다운로드 토큰/만료. 클라이언트 지정 id는 UUID 강제 + `Persistable`로 진짜 INSERT(중복 409)
- **`stored_files`**: 업로드 이미지 바이트(LONGBLOB). `APP_STORAGE_TYPE=db`(운영 기본)일 때 사용 —
  컨테이너 재배포에도 보존. `local`(개발)이면 `./uploads/` 디스크

### 4.5 배포 안전장치 (RailwayDeploymentValidator)

`RAILWAY_PROJECT_ID` 감지 시:
- JDBC URL이 비었거나 H2면 **부팅 실패** (인메모리 데이터 소실 방지)
- JWT 시크릿이 기본값이면 **부팅 실패**
- `APP_PUBLIC_BASE_URL` 미설정이면 WARN

## 5. Flutter 앱 (businesscard_qr_app)

- base URL: `.env`(`BACKEND_BASE_URL`) → 기본값도 businesscard_qr 자기 도메인. localhost 아님
- 네트워크: 전 호출 타임아웃(20s/업로드 60s), 401→refresh→재시도 1회, 502/503/504 "서버 깨어나는 중" 분기
- 업로드: 확장자→Content-Type 명시(jpg/png/gif/bmp/webp), HEIC 사전 차단, `imageQuality:85`
- 서버 배포와 무관 (Docker 이미지에 미포함). APK 별도 빌드/배포
