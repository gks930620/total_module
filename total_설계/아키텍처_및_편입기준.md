## 1. 확정 방향 (한 장 요약)

```
GitHub 저장소 1개 (total_module)  =  Railway 프로젝트 1개
빌드: 루트 단일 Dockerfile → 이미지 1개 (모든 모듈 jar 포함)
실행: 모듈 1개 = Railway 서비스 1개 (APP_MODULE로 역할 선택)
접근: 각 서비스가 자기 공개 URL로 직접 트래픽 받음 — ⚠️ 게이트웨이 없음
```

| 항목 | 결정 |
|------|------|
| 저장소 / Railway 프로젝트 | **각 1개 고정.** 모듈이 늘어도 서비스만 추가 |
| 실행 단위 | **모듈 = 서비스 (1:1).** jar 하나로 합치지 않는다 |
| 외부 접근 | **각 서비스 자기 공개 URL 직행.** 게이트웨이/프록시 없음 |
| 빌드 | **루트 단일 Dockerfile + 멀티스테이지.** APP_MODULE로 실행 모듈 선택 |
| DB | **공통 MySQL 1개 + 모듈별 스키마 분리** (비용) — [`railway_배포_및_비용.md`](railway_배포_및_비용.md) |
| dist_api_gateway | **은퇴·삭제 완료** (§6 이력) |

## 2. 왜 이렇게 결정했나

**모듈=서비스 (jar 하나로 안 합침)**
- 컨셉이 "로컬에서 완성한 프로젝트를 그대로 가져와 배포". 진짜 모놀리스로 합치면 가져올 때마다 **병합 수술**(SecurityConfig/yml/빈이름/의존성 충돌) + 한 모듈 장애가 전체 전파 + 전체 재배포.
- 모듈=서비스면 편입 비용이 **등록 절차 + env 몇 개로 고정**되고 모듈끼리 충돌할 수 없다.
- (트래픽이 커진 특정 모듈만 따로 떼거나, 반대로 비용상 하나로 합치는 건 나중에 선택 가능.)

**게이트웨이 없음**
- 게이트웨이는 초기 "서버 1대로 합쳐 배포" 설계의 유산. 서비스 분리로 바뀐 순간 각 서비스가 공개 URL을 가져 존재 이유 소멸.
- 순수 리버스 프록시는 얻는 것보다 잃는 게 컸다: 절대 URL이 X-Forwarded에 의존해 깨지고, 리다이렉트/에러코드/스트리밍/타임아웃을 직접 재구현. 새 모듈마다 라우팅·프리픽스 공사.
- 필요해지면 그때 **검증된 것**(Spring Cloud Gateway/Nginx)을 인프라 레벨로 붙인다(역방향이 어려움).

**루트 단일 Dockerfile (모듈별 Dockerfile 아님)**
- 모듈들이 같은 Java/Spring 스택 + 공통 설정이라, 인프라 변경(Java 버전 등)을 **한 파일에서** 관리. 멀티스테이지로 공통 레이어 캐시.
- 모듈이 10개가 돼도 Dockerfile은 1개, Railway 서비스만 추가하고 `APP_MODULE`만 바꾼다.
- (스택이 완전히 다른 모듈-예: Go/Node-이 생기면 그때만 그 모듈용 별도 Dockerfile 고려.)

**Railway 프로젝트 1개**
- 같은 프로젝트 안 서비스끼리 내부 네트워크·env 참조(`${{service.VAR}}`) 가능, 대시보드 일원화. 새 모듈 = 서비스 추가이지 프로젝트 추가가 아니다.

## 3. 저장소 구조

```
/total_module (상위 폴더 — 실행 코드 없음, 스프링부트 아님)
  ├── Dockerfile            # 전체 조립 설계도 — 모든 모듈 jar를 이미지 하나에 (start.sh 내장)
  ├── railway.toml          # Railway 빌드/헬스체크(/healthz)
  ├── settings.gradle       # 하위 모듈 명단 (include 'businesscard_qr' 등)
  ├── build.gradle          # 공통 규칙 (거의 비어있음)
  │
  ├── /businesscard_qr      # [모듈] 명함 QR API (Spring Boot — 독립 앱이면서 모듈)
  ├── /doll_gacha           # [모듈] 인형뽑기 (Spring Boot + React 정적 서빙)
  ├── /project_a            # [모듈] 향후 추가...
  │
  ├── /businesscard_qr_app  # Flutter 앱 — Gradle 멤버 아님(같이 두고 보는 별개 프로젝트)
  ├── /total_설계            # 전체 설계 문서
  └── /클로드코드와응답        # 작업 기록 (Claude 산출물 — 역사)
```

- 각 스프링부트 모듈은 **독립 실행 가능한 완전한 앱**(자기 main·jar·gradlew).
- 게이트웨이/현관문 모듈은 두지 않는다.

## 4. 새 프로젝트 편입 절차

> 전제: 프로젝트는 **로컬에서 완성**해서 가져온다. 여기서는 정리 + 배포 적합화만.
> 편입 = **git 관리 주체가 total_module로 넘어옴.** 원 저장소는 "이관 전 히스토리 보관소"로만 남긴다.

### 4-1. 가져오기 전 정리 (순서 중요)
- [ ] **원 저장소에 push 안 된 커밋 없는지 확인** → 있으면 먼저 push (히스토리 보존)
  `git log --oneline @{u}..HEAD | wc -l` 가 0이어야 안전
- [ ] **프로젝트 내부 `.git` 폴더 삭제** ⚠️ 안 지우면 gitlink(중첩 저장소)로 커밋돼 clone 시 내용이 통째로 빠진다(doll_gacha에서 실제 발생).
- [ ] **클라이언트 앱 분리**: 앱/웹 클라이언트는 최상위 별도 폴더로(Gradle include 대상 아님). 필요 없으면 삭제.
- [ ] **죽은 리소스 제거**: 안 쓰는 templates/·옛 프로파일 yml·data-*.sql 등.
- [ ] **비밀값 점검**: `.env`·yml에 커밋된 시크릿 없는지. 운영 값은 전부 Railway env로. `.env`는 커밋 금지.
- [ ] **로컬 전용 인프라 파일**: 프로젝트 자체 Dockerfile/compose는 Railway 배포에 안 쓰임 → 남기려면 상단에 `# 로컬 개발 전용` 주석.

### 4-2. 빌드 특이사항 ⚠️
- [ ] **프론트 빌드(npm/vite)가 gradle 태스크에 묶였는지** 확인. 루트 Docker 빌드엔 **node가 없어서** 그대로면 이미지 빌드가 깨진다.
  → 대응: 빌드 산출물을 `static/`에 커밋 + npm 태스크는 `onlyIf`(npm 있을 때만)로 가드 = 로컬은 자동빌드, Docker는 커밋된 산출물 사용. (doll_gacha 사례)

### 4-3. 배포 적합화 (환경 무관 실행 — 상세는 [`코드컨벤션.md`](코드컨벤션.md) §5)
- [ ] DB 등 설정이 **env 주입** (`${SPRING_DATASOURCE_URL:로컬기본값}` 패턴)
- [ ] `server.port: ${PORT:...}` / **`GET /healthz`**(200 `{status:UP}`)
- [ ] 파일 저장이 로컬 디스크가 아니라 **Storage Bucket**(운영)/디스크 폴백(로컬) — 컨벤션 §5-3
- [ ] 프로파일: Railway에서 어떤 프로파일로 뜨는지 명확히(`SPRING_PROFILES_ACTIVE=prod`)
- [ ] 운영에서 인메모리/H2로 부팅되면 **fail-fast**(RailwayDeploymentValidator)

### 4-4. 등록 (코드)
1. 폴더를 루트에 투하
2. `settings.gradle`에 `include '<module>'`
3. 루트 `Dockerfile`의 `[새 모듈 추가 시]` 3곳(COPY·bootJar·jar추출) + start.sh `APP_MODULE` 허용 목록에 추가
4. → Railway 서비스 생성은 [`railway_배포_및_비용.md`](railway_배포_및_비용.md)

### 4-5. Git 규칙
- 편입/작업은 **브랜치 → PR 리뷰 → master 머지**. **master push = Railway 재배포 트리거**이므로 배포 env 준비 후 push.

## 5. 웹 프로젝트 유형별 기준

> 접근은 무조건 **자기 공개 URL 직행**(고민 없음). 화면이 어디서 나오냐만 판단.

| 유형 | 화면(HTML/JS) | 편입 시 |
|------|------|------|
| **1. 정적웹 분리형** (CSR + API 분리) | 별도 정적 호스팅(github.io/Vercel) | **백엔드만** 모듈 편입 + 화면 서빙 잔재 제거 + **CORS(정적 오리진)** + **인증방식 점검** |
| **2. 스프링 일체형** | 스프링부트가 직접 서빙(@Controller+정적) | **통째로 모듈 편입.** CORS 불필요, 세션쿠키도 그냥 됨 |

- **네이티브 앱**(명함QR)은 유형 1과 같되 CORS 불필요.
- ⚠️ **유형 1 인증**: JWT 토큰이면 OK. **세션+쿠키면** 교차 도메인(github.io↔railway.app) SameSite로 쿠키가 안 붙음 → 토큰 방식 전환 필요.
- doll_gacha는 React지만 **스프링부트가 정적 서빙**(일체형에 가까움) → 같은 오리진이라 마찰 적음.

## 7. 부록 — 다음 프로젝트를 새로 만든다면 (교훈)

이번 프로젝트에서 실제로 아팠던 것들. 다음엔 처음부터 피하고 시작.

**설계**
- **게이트웨이 프록시 자작 금지** — 앱이 API를 직접 호출. 정말 필요하면 검증된 Cloud Gateway/Nginx.
- **공개 URL을 요청 헤더에서 유추 금지** — 처음부터 `APP_PUBLIC_BASE_URL`(또는 `APP_BASE_URL`) env로 명시, 모든 절대 URL을 여기서 조립. `forward-headers-strategy: framework`.
- **운영에서 인메모리 DB면 즉시 죽게(fail-fast)** — "조용한 성공"보다 "시끄러운 실패". `DeploymentValidator`를 처음부터 + 필수 env 한곳 관리.

**스프링부트** (상세 규칙은 [`코드컨벤션.md`](코드컨벤션.md))
- 도메인별 패키지(`auth`/`user`/`<domain>`/`common`), `open-in-view:false`, 소프트삭제(`is_active`).
- 전역 예외는 `ResponseEntityExceptionHandler` 상속(표준 4xx 보존). 외부 API 호출은 트랜잭션 밖 + 타임아웃. 카운터는 원자적 UPDATE. 파일 업로드 Content-Type 화이트리스트 + 413 핸들러.
- JWT: access(짧게)+refresh(회전). 통합테스트(컨트롤러 레벨)만.

**Flutter 앱** (이번에 가장 부족했던 부분)
- 단일 `ApiClient`로 일원화, **모든 호출 타임아웃(15~30s)**, **401→refresh 1회→실패 시 재로그인** 인터셉터, **5xx/비-JSON 분기**("서버 깨우는 중" UX — 콜드스타트 대응).
- base URL은 `.env`/`--dart-define`(기본값도 운영 HTTPS, localhost 금지). 토큰은 `shared_preferences`/secure storage. `.env` 커밋 금지.
- JSON 계약(snake_case, 멀티파트 파트명, 응답 봉투)을 문서로 고정하고 양쪽 참조.
