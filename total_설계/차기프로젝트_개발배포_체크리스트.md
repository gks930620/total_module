# 파트 A. 다음 프로젝트를 새로 만든다면 (앱 + 스프링부트)

## A0. 이번 프로젝트에서 얻은 핵심 교훈 (먼저 읽기)

이번 구조("앱 ↔ 게이트웨이 ↔ 핵심 API ↔ DB")에서 실제로 아팠던 지점들. 다음 프로젝트는 **이걸 처음부터 피하고 시작**하면 된다.

1. **게이트웨이 프록시는 얻는 것보다 잃는 게 많았다.**
   순수 리버스 프록시 한 모듈을 두니 → 절대 URL(QR/이미지)이 X-Forwarded 헤더에 의존해 깨지기 쉬웠고, 리다이렉트/에러 상태코드/스트리밍/타임아웃을 전부 직접 재구현해야 했다.
   → **다음엔 앱이 핵심 API를 직접 호출**. 프록시가 정말 필요할 때만(여러 백엔드 통합 등) 도입하고, 그땐 Spring Cloud Gateway 같은 **검증된 프록시**를 쓴다. 직접 만든 RestTemplate 프록시는 지양.

2. **"공개 URL"을 요청 헤더에서 유추하지 말 것.**
   `ServletUriComponentsBuilder.fromCurrentContextPath()`로 절대 URL을 만들면 프록시/로드밸런서 뒤에서 내부 호스트가 새어 나온다.
   → 처음부터 **`APP_PUBLIC_BASE_URL` 같은 환경변수로 공개 base URL을 명시**하고, 모든 절대 URL을 여기서 조립.

3. **DB 설정 실수는 "조용히" 일어난다.**
   운영 DB env를 빠뜨리면 H2 인메모리로 조용히 부팅 → 재배포마다 데이터 소실.
   → **운영에서 인메모리 DB로 부팅되면 즉시 죽게(fail-fast)** 만든다. "조용한 성공"보다 "시끄러운 실패"가 낫다.

4. **앱은 "느리고 불안정한 네트워크"를 기본 가정으로.**
   Railway 무료 티어는 콜드스타트(수 초~수십 초)와 재배포 중 5xx가 일상.
   → 앱의 모든 네트워크 호출에 **타임아웃 + 재시도 + 401/5xx 분기**를 처음부터 넣는다. (이번엔 이게 없어서 무한 로딩)

5. **토큰은 앱에 영구 저장 + 서버 신호로 갱신.**
   JWT를 메모리에만 캐시하고 로컬 시계로만 만료 추정하면, 서버 재배포로 시크릿이 바뀔 때 복구가 안 된다.
   → `shared_preferences`(또는 secure storage)에 저장, **401 받으면 refresh → 실패 시 재로그인** 흐름을 표준으로.

6. **에러 상태코드를 뭉개지 말 것.**
   전역 `Exception` 핸들러가 표준 4xx를 500으로 바꾸면 클라이언트/모니터링이 오작동.
   → `ResponseEntityExceptionHandler`를 상속해 프레임워크 예외의 상태코드를 보존.

## A1. 권장 아키텍처 (다음 프로젝트 기본형)

```
[Flutter 앱]
   │  HTTPS + Authorization: Bearer <JWT>
   │  (타임아웃/재시도/401→refresh 내장)
   ▼
[Spring Boot 단일 API 서버]  ← Railway에 1개 서비스로 배포
   │   · 인증(소셜 로그인 → 자체 JWT)
   │   · 도메인 API (REST, snake_case 계약)
   │   · 파일은 DB(LONGBLOB) 또는 S3/오브젝트 스토리지
   │   · 공개 URL은 APP_PUBLIC_BASE_URL로 조립
   ▼
[관리형 DB]  ← Railway MySQL/Postgres 또는 외부 관리형 DB
```

- **게이트웨이 모듈 없음**(위 교훈 1). 앱이 API 서버를 직접 호출.
- 나중에 정말 통합 게이트웨이가 필요하면 그때 **별도 인프라 레벨**(Cloud Gateway/Nginx)로 추가.

## A2. 스프링부트 쪽 개발 체크리스트

**프로젝트 초기 세팅**
- [ ] Java 17+ / Spring Boot 3.x, 도메인별 패키지(`auth`, `user`, `<domain>`, `common`).
- [ ] `open-in-view: false`, 소프트 삭제(`is_active`) 등 이번에 잘 된 관례 유지.
- [ ] `application.yml`의 **모든 운영 설정을 환경변수로** 노출(기본값은 로컬 개발용).

**배포 안전장치 (fail-fast) — 이번 프로젝트의 가장 큰 교훈**
- [ ] 운영(예: `RAILWAY_PROJECT_ID` 존재) 감지 시 아래를 **부팅 실패**로 강제:
  - JDBC URL이 H2/인메모리면 → 죽기 (데이터 소실 방지)
  - JWT 시크릿이 기본값이면 → 죽기
  - `APP_PUBLIC_BASE_URL`이 비었거나 localhost면 → 죽기
- [ ] 이 검증기(`DeploymentValidator`)를 처음부터 만들고, **필수 env 목록을 한곳에서 관리**.

**API/보안**
- [ ] 절대 URL은 `APP_PUBLIC_BASE_URL` 기반으로만 생성(요청 헤더 유추 금지).
- [ ] 전역 예외처리는 `ResponseEntityExceptionHandler` 상속 + 커스텀 예외로 상태코드 명확히(문자열 매칭 금지).
- [ ] 외부 API 호출(소셜 로그인 등)은 **트랜잭션 밖**에서 + connect/read 타임아웃 필수.
- [ ] 카운터 등 동시성 필드는 **원자적 UPDATE 쿼리**로.
- [ ] 파일 업로드: Content-Type 화이트리스트 + 크기초과 413 핸들러. 저장은 DB LONGBLOB이면 `max_allowed_packet` 주의, 규모 커지면 S3.
- [ ] JWT: access(짧게) + refresh(저장소 기반 회전) — 진짜 폐기가 필요하면 jti 화이트리스트/버전 컬럼.

**테스트/문서**
- [ ] 통합 테스트(이번 `ApiIntegrationTest`처럼) 유지. 프록시를 쓴다면 **와이어 레벨**(MockWebServer)로.
- [ ] `/healthz` 헬스체크, Swagger.

## A3. Flutter 앱 쪽 개발 체크리스트

**네트워크 계층 (이번에 가장 부족했던 부분)**
- [ ] 단일 `ApiClient`로 모든 호출 일원화(직접 `http.get` 흩뿌리지 말 것).
- [ ] **모든 호출에 `.timeout(15~30s)`**.
- [ ] **401 → 토큰 refresh 1회 → 실패 시 로그아웃/재로그인** 인터셉터.
- [ ] **5xx / 비-JSON 응답 분기**: "서버 깨우는 중, 재시도" UX(콜드스타트 대응).
- [ ] 응답 파싱 전에 status/Content-Type 확인(모든 본문을 JSON으로 가정 금지).

**설정/보안**
- [ ] base URL은 `--dart-define` 또는 `.env`로 주입, **기본값도 운영 HTTPS**(localhost 금지).
- [ ] 비밀키(소셜 앱 키 등)는 `.env`를 **git에 커밋하지 말 것**(이번엔 커밋됨). 릴리스는 CI 시크릿/`--dart-define`.
- [ ] 토큰은 `shared_preferences`(또는 `flutter_secure_storage`)에 저장.
- [ ] `usesCleartextTraffic`는 두지 않기(HTTPS만). 시작 시 필수 env는 강제 언랩(`!`) 대신 검증 후 친절한 에러.
- [ ] async 후 `context` 사용 시 `if (!mounted) return;` 가드.

**계약(앱↔서버 일치) — 이번엔 잘 됐던 부분, 유지**
- [ ] JSON 키 규칙(snake_case), 멀티파트 파트명, 응답 봉투(`{success,data,message}`)를 **문서로 고정**하고 양쪽이 참조.

# 파트 B. Railway 배포 — 사용자(나)가 직접 준비/입력해야 하는 것

> ⚠️ **아래 값들은 코드가 자동으로 알 수 없다. 배포 전에 "사람"이 준비해서 Railway에 넣어야 한다.**
> 하나라도 빠지면 조용히 잘못 동작하거나(→ 데이터 소실) 부팅에 실패한다.

## B1. 배포 전에 사람이 미리 만들어 와야 하는 것 (계정/외부 리소스)

| 준비물 | 어디서 | 왜 필요 / 메모 |
|--------|--------|----------------|
| **Railway 프로젝트 + DB 플러그인** | Railway 대시보드에서 MySQL/MariaDB 추가 | 관리형 DB. 여기서 접속 URL/계정을 얻어 아래 env에 넣음 |
| **카카오 개발자 앱** | developers.kakao.com | REST 검증용은 서버가 `/v2/user/me` 호출로 처리하지만, **앱용 네이티브 키·리다이렉트 스킴**은 사람이 발급/등록 |
| **JWT 시크릿 문자열** | 직접 생성(랜덤 32바이트+) | `openssl rand -base64 48` 등으로 만들어 둘 것. 소스 기본값 쓰면 배포 차단됨 |
| **앱이 바라볼 공개 도메인** | Railway가 배포 후 부여(`*.up.railway.app`) 또는 커스텀 도메인 | 앱 base URL + 서버 `APP_PUBLIC_BASE_URL`에 사용 |

## B2. Railway 환경변수 — businesscard_qr(핵심 API 서비스)

> 서비스 하나를 만들고 아래를 넣는다. **★ 표시는 없으면 데이터 소실/부팅 실패로 이어지는 필수값.**

| 환경변수 | 필수 | 예시 / 값 | 사람이 정해야 하는 이유 |
|----------|:---:|-----------|------------------------|
| `APP_MODULE` | ★ | `businesscard_qr` | 어느 jar을 띄울지 (미설정 기본값도 businesscard_qr) |
| `SPRING_DATASOURCE_URL` | ★ | `jdbc:mariadb://<host>:<port>/<db>` | **빠뜨리면 H2 인메모리로 조용히 떠서 재배포마다 데이터 소실** (문제 D1). Railway DB 플러그인 값에서 조립 |
| `SPRING_DATASOURCE_USERNAME` | ★ | (DB 플러그인 값) | DB 접속 계정 |
| `SPRING_DATASOURCE_PASSWORD` | ★ | (DB 플러그인 값) | DB 접속 비번 |
| `SPRING_DATASOURCE_DRIVER` | ○ | `org.mariadb.jdbc.Driver` | MariaDB URL이면 넣어줄 것(기본값이 H2라 안 넣으면 부팅 실패 가능, 문제 B3) |
| `APP_JWT_SECRET` | ★ | 랜덤 긴 문자열 | 소스 기본값이면 배포 차단됨. 사람이 생성 |
| `APP_PUBLIC_BASE_URL` | ★ | `https://<해당 서비스의 자기 공개 도메인>` | QR/이미지/다운로드 절대 URL의 기준. ✅ 2026-07-08 코드에 도입 완료. 게이트웨이 은퇴 방향에 따라 **자기 도메인**을 넣는다 |
| `APP_CORS_ALLOWED_ORIGINS` | △ | 웹 클라이언트 도메인들 | 네이티브 앱만이면 불필요. 웹 있으면 필수 |
| `APP_STORAGE_TYPE` | ○ | `db` (기본) | 이미지 저장 방식. `db`면 DB packet 크기 확인 |
| `SPRING_JPA_DDL_AUTO` | ○ | 초기 `update`, 안정화 후 `validate` | 스키마 자동변경 정책. 운영 안정기엔 `validate` 권장 |
| `SPRING_H2_CONSOLE_ENABLED` | ○ | (설정 안 함=false) | 운영에서 절대 true 금지 |

## B3. ~~Railway 환경변수 — dist_api_gateway~~ (2026-07-08 폐기 확정)

## B4. 앱(Flutter) 쪽에서 사람이 넣어야 하는 값

| 값 | 어디에 | 메모 |
|----|--------|------|
| 서버 base URL | `.env`의 `BACKEND_BASE_URL` 또는 `--dart-define` | **API 서버 자기 공개 HTTPS 도메인.** localhost 금지 |
| 카카오 네이티브 앱 키 | `.env`의 `KAKAO_NATIVE_APP_KEY` + AndroidManifest 리다이렉트 스킴 | 카카오 콘솔에서 발급. `.env`는 **커밋 금지**(현재 커밋되어 있으니 정리 필요) |

## B5. 배포 직후 사람이 눈으로 확인할 것 (체크리스트)

- [ ] **DB가 진짜 MariaDB인가?** (H2로 뜨지 않았는지 로그/데이터 영속 확인) — 문제 D1
- [ ] 앱에서 명함 생성 → **이미지 URL/QR URL이 `https://<공개도메인>`으로 시작**하는지 실기기 확인 — 문제 D2
- [ ] 재배포 후에도 기존 회원·명함·이미지가 **그대로 남아 있는지**
- [ ] 콜드스타트(한동안 방치 후 첫 요청) 시 앱이 **무한 로딩에 빠지지 않는지** — 문제 D3/D4
- [ ] `/healthz` 200, Swagger 접근, H2 콘솔은 **닫혀 있는지**
