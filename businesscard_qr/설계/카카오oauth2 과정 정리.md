# 카카오 OAuth2 과정 정리

기준일: 2026-04-22

## 1) 현재 프로젝트 구현 방식
1. 앱이 카카오 SDK로 로그인/동의 화면을 처리하고 카카오 `access_token`을 받는다.
2. 앱이 카카오 토큰을 백엔드 `POST /api/auth/kakao`로 전달한다.
3. 백엔드는 카카오 `user/me`를 호출해 토큰 유효성과 사용자 정보를 검증한다.
4. 백엔드 DB에서 사용자 정보를 `upsert`한다. (있으면 갱신, 없으면 생성)
5. 백엔드가 자체 JWT(access token)를 발급한다.
6. 앱은 이후 `Authorization: Bearer <자체 JWT>`로 `/api/**`를 호출한다.

핵심 요약: `OAuth2(카카오) + 토큰 검증 백엔드 + 자체 JWT 세션` 패턴

## 2) OAuth2 용어로 매핑
- 앱 ↔ 카카오 로그인 단계: 보통 Authorization Code 기반(모바일 SDK가 내부 처리, PKCE 포함 가능)
- 앱이 받은 카카오 토큰을 백엔드에 전달하고, 백엔드가 재검증하는 단계: Token Relay / Backend Verification 패턴
- 백엔드가 최종적으로 자체 JWT를 발급하는 단계: First-party JWT(자체 인증 체계)

## 3) 백엔드 주도 방식(대안)
정식 명칭: `OAuth2 Authorization Code Grant (Server-side / Confidential Client)`

흐름:
1. 앱/웹이 로그인 시작 요청을 백엔드로 보낸다.
2. 백엔드가 카카오 인증 페이지로 리다이렉트한다.
3. 카카오가 Authorization Code를 백엔드로 콜백한다.
4. 백엔드가 Code로 카카오 토큰을 교환한다.
5. 백엔드가 사용자 정보 조회/DB 동기화 후 자체 JWT를 발급한다.

차이점: 카카오 토큰 취급 지점을 앱이 아니라 백엔드 중심으로 더 모을 수 있다.

## 4) 현재 방식과 대안 방식 비교
| 항목 | 현재 방식 (앱 토큰 전달) | 백엔드 주도 방식 (Code 교환) |
|---|---|---|
| 카카오 토큰 최초 수신 | 앱 | 백엔드 |
| 백엔드의 카카오 검증 | 수행 (`user/me`) | 수행 (토큰 교환 + 사용자 조회) |
| 앱 복잡도 | 낮음 (SDK 중심) | 상대적으로 높음 (리다이렉트/콜백 처리 고려) |
| 토큰 통제력 | 중간 | 높음 |
| 우리 API 인증 | 자체 JWT(Bearer) | 자체 JWT(Bearer) |

## 5) 이 프로젝트의 권장 운영 포인트
- 외부 제공자(카카오) 토큰은 로그인 직후 검증 용도로만 사용한다.
- 앱이 비즈니스 API를 호출할 때는 카카오 토큰이 아니라 자체 JWT만 사용한다.
- `authenticated()` 보호 API는 `Authorization: Bearer <자체 JWT>`만 통과시킨다.
- 외부 공개 링크(예: QR 다운로드)는 필요한 최소 경로만 `permitAll`로 유지한다.
