# 구글플레이 배포 후 결제/정산 실행 가이드 (개인사업자 기준)

## 0) 결론 먼저
- **Play 배포 앱에서 디지털 상품/기능을 앱 내에서 판매**하면 기본적으로 **Google Play 결제**를 써야 합니다.
- **카카오페이(카톡결제)를 앱 내부 결제로 바로 쓰는 건 기본 정책상 불가**이며, 예외는 Google의 **대체결제 프로그램 등록 + 요구 API/UX 구현**이 필요합니다.
- 돈이 실제로 계좌로 들어오려면:
  1. Play Console + Payments profile(판매자) 설정
  2. 세금 정보 입력
  3. 정산용 은행계좌 등록/검증/Primary 지정
  4. 인앱상품 생성/출시
  5. 앱 결제 코드 + 백엔드 검증/승인(ack) 구현

---

## 1) 결제수단 선택 기준 (중요)

### 1-1. Google Play 결제 의무
Play 정책상, Play 배포 앱에서 인앱 디지털 기능/콘텐츠를 판매하면 Google Play 결제를 사용해야 합니다.

### 1-2. 카카오페이(대체결제) 가능한 경우
- 한국은 **User Choice Billing(유저 선택 결제)** 프로그램이 별도로 존재합니다.
- 이 경우도 그냥 카카오페이 버튼만 넣는 게 아니라, **Google이 요구하는 대체결제 API/온보딩/UX**를 구현해야 합니다.

### 1-3. 물리 상품/오프라인 서비스
- 음식배달, 택시, 실물배송 등은 Google Play 결제 대상이 아니어서 일반 PG(카카오페이 포함) 사용 가능.

---

## 2) 돈이 사업자 계좌로 들어오기까지 필수 설정

## 2-1. Payments profile 생성/연결
1. Play Console 로그인
2. `Settings > Payments settings`
3. **Create payments profile**
4. 법적 상호, 주소, 고객센터 정보, 명세서 표시명 입력

주의:
- Payments profile 국가와 정산 계좌 국가는 일치해야 합니다.

## 2-2. 세금 정보 입력
`Settings > Payments settings > [country] tax info`에서 세금정보 입력.

## 2-3. 정산 계좌 등록/검증
1. `Settings > Payments settings > How you get paid`
2. 계좌 추가
3. 계좌 검증(국가/방식에 따라 즉시 또는 소액입금 방식)
4. **Primary 계좌 지정**

## 2-4. 정산 주기
- Google Play 개발자 정산은 일반적으로 **매월 15일에 전월 매출분 지급 시작**.
- 실제 입금 반영은 EFT/송금 방식에 따라 며칠 더 걸릴 수 있음.

---

## 3) Play Console 상품/테스트 준비

1. 앱을 내부 테스트 트랙이라도 업로드(결제 기능 활성화에 필요)
2. 인앱상품 생성
   - One-time product(관리형 상품)
   - Subscription(구독)
3. 라이선스 테스터 설정
   - `Settings > License testing`
4. 테스트 계정으로 결제 시나리오 확인

## 4) 앱 코드 (Flutter) 기본 패턴

아래는 `in_app_purchase` 기준 최소 패턴입니다.

```dart
import 'dart:async';
import 'package:in_app_purchase/in_app_purchase.dart';

class BillingService {
  final _iap = InAppPurchase.instance;
  late final StreamSubscription<List<PurchaseDetails>> _sub;

  Future<void> init() async {
    final available = await _iap.isAvailable();
    if (!available) throw Exception('Play 결제 사용 불가');

    _sub = _iap.purchaseStream.listen(_onPurchaseUpdated);
  }

  Future<ProductDetails> loadProduct(String productId) async {
    final resp = await _iap.queryProductDetails({productId});
    if (resp.notFoundIDs.isNotEmpty || resp.productDetails.isEmpty) {
      throw Exception('상품 조회 실패: $productId');
    }
    return resp.productDetails.first;
  }

  Future<void> buy(ProductDetails product) async {
    final param = PurchaseParam(productDetails: product);
    // 소모성은 buyConsumable, 비소모성/구독은 buyNonConsumable
    await _iap.buyNonConsumable(purchaseParam: param);
  }

  Future<void> _onPurchaseUpdated(List<PurchaseDetails> purchases) async {
    for (final p in purchases) {
      if (p.status == PurchaseStatus.purchased || p.status == PurchaseStatus.restored) {
        // 1) purchaseToken 등 서버로 전달
        final ok = await _verifyOnBackend(
          productId: p.productID,
          purchaseToken: p.verificationData.serverVerificationData,
        );
        if (ok && p.pendingCompletePurchase) {
          // 2) 검증 + 권한부여 완료 후 completePurchase
          await _iap.completePurchase(p);
        }
      } else if (p.status == PurchaseStatus.error) {
        // 에러 처리
      }
    }
  }

  Future<bool> _verifyOnBackend({
    required String productId,
    required String purchaseToken,
  }) async {
    // TODO: 백엔드 /billing/google/verify 호출
    return true;
  }

  Future<void> dispose() async {
    await _sub.cancel();
  }
}
```

핵심:
- 결제 성공 직후 **클라이언트 단독 신뢰 금지**
- 반드시 백엔드 검증 후 권한부여
- `completePurchase`/ack가 늦으면 환불될 수 있으므로 빠르게 처리

## 5) 백엔드 코드 (Spring) 기본 패턴

## 5-1. 앱 -> 서버 검증 요청 DTO
```java
public record GooglePurchaseVerifyRequest(
        String packageName,
        String productId,
        String purchaseToken,
        boolean subscription,
        String appUserId
) {}
```

## 5-2. 컨트롤러
```java
@RestController
@RequestMapping("/api/billing/google")
public class GoogleBillingController {

    private final GoogleBillingService googleBillingService;

    public GoogleBillingController(GoogleBillingService googleBillingService) {
        this.googleBillingService = googleBillingService;
    }

    @PostMapping("/verify")
    public Map<String, Object> verify(@RequestBody GooglePurchaseVerifyRequest req) {
        var result = googleBillingService.verifyAndGrant(req);
        return Map.of("success", true, "data", result);
    }
}
```

## 5-3. 서비스 핵심 흐름
```java
@Service
public class GoogleBillingService {

    public VerifyResult verifyAndGrant(GooglePurchaseVerifyRequest req) {
        // 1) DB에서 purchaseToken 중복 체크 (토큰은 유니크 키 권장)
        // 2) Google Play Developer API 조회:
        //    - one-time: purchases.products.get
        //    - subs: purchases.subscriptionsv2.get
        // 3) 상태가 PURCHASED/ACTIVE인지 확인 (PENDING이면 권한부여 금지)
        // 4) 정상 구매면 사용자 권한(entitlement) 부여
        // 5) acknowledge/consume 호출
        // 6) 처리 결과 DB 기록
        return new VerifyResult(true, "OK");
    }
}
```

## 5-4. Google API 호출 대상 (서버)
- One-time 검증  
  `GET /androidpublisher/v3/applications/{packageName}/purchases/products/{productId}/tokens/{token}`
- Subscription 검증  
  `GET /androidpublisher/v3/applications/{packageName}/purchases/subscriptionsv2/tokens/{token}`
- 승인(ack)/소모(consume)도 서버에서 처리 권장

## 6) 운영 안정화 필수

1. **RTDN(Real-time Developer Notifications)** 구성  
   - Pub/Sub 토픽 + 권한 + Play Console 연동
2. RTDN 수신 시 Google API 재조회 후 상태 동기화
3. 취소/환불/갱신실패 반영 자동화
4. 토큰 중복처리, idempotency 처리
5. 주문/정산 리포트 주기 검증

## 7) 카카오페이 쓰고 싶다면 (현실적인 가이드)

### 선택지 A: 가장 단순/안전
- Play 배포 앱 인앱 디지털 결제는 **Google Play 결제만 사용**.

### 선택지 B: 한국 대체결제(유저선택결제) 참여
- Play Console에서 대체결제 프로그램 온보딩
- 요구 API/안내 화면/신고(리포팅) 구현
- 서비스 수수료 구조와 운영 복잡도 증가

즉, 처음 배포/수익화 단계에서는 A가 구현과 리스크 측면에서 훨씬 안정적입니다.
