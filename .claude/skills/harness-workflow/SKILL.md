# 하네스 오케스트레이션 가이드

메인 세션(지휘자)이 **역할 에이전트**를 어떻게 부리는지에 대한 규칙이다.
핵심은 **"AI 한 명한테 다 시키지 않는다"** — 회사 조직처럼 역할별로 넘긴다.
팀 명단·수정권한·전체 파이프라인은 `.claude/CLAUDE.md` 참조. 이 스킬은 "실제로 어떻게 굴리나".

## 언제 팀을 쓰나 (판단 기준)

| 상황 | 처리 |
|---|---|
| 한두 줄 수정, 단순 질문, 탐색("어디 있지?") | **팀 안 씀.** 메인 세션이 그냥 처리 (탐색은 Explore 서브에이전트) |
| 여러 파일·여러 스택이 걸린 기능 | **풀 파이프라인** (기획→설계→구현→검증) |
| 계약/설계는 정해졌고 구현만 | senior-dev 건너뛰고 **해당 개발자(backend/frontend/app-dev)부터** |
| 구현 끝, 동작만 검증 | **qa만** 호출 |
| 스택에 안 걸리는 잡일(스크립트·설정·문서) | **builder** |

> 과하게 쓰지 마라. 팀을 나누는 건 공짜가 아니다(컨텍스트·시간). **복잡할 때만** 나눈다.

## 기본 흐름 (풀 파이프라인) — TDD 가 핵심 규칙

```
product-planner(사용자 흐름·인수 조건) → designer(화면 정의서)
→ senior-dev(API 명세 도출 + ERD + 명세 기반 테스트 선작성 = Red)
→ backend-dev ∥ frontend-dev ∥ app-dev  (테스트 통과시키며 병렬 구현 = Green)
→ senior-dev(코드리뷰 — 컨벤션·스택 간 정합성, 수정은 개발자가)
→ qa(기획서로 테스트케이스 설계·시나리오 시험) — 결함은 해당 개발자로 되돌림
→ devops(커밋·배포)
```

1. **product-planner** — 사용자 관점 동작 흐름·인수 조건을 받는다. (문서만 수정)
2. **designer** — 화면 정의서(상태별 UI). (문서·토큰만)
3. **senior-dev** — 기획서에서 **API 계약·ERD 도출 + 실패하는 테스트를 먼저 작성(Red)**. 테스트 없이 개발자에게 넘기지 않는다.
4. **backend/frontend/app-dev** — 각 스택에서 **테스트를 통과시키는 구현(Green)**. senior-dev 의 테스트를 고치지 않는다(틀렸으면 되돌림). 자기 영역 밖 스택 금지.
5. **senior-dev(리뷰)** — 컨벤션·클린코드·스택 간 정합성 점검. **수정은 원 개발자가** 한다(verify 루프).
6. **qa** — 기획서로 테스트케이스를 설계해 실제 제품을 시험(경계·비정상·탐색·회귀). 결함은 해당 개발자에게 되돌림. **qa 통과가 완료 조건.** (코드 수정 권한 없음)
7. **devops** — 커밋·배포 (사용자가 명시적으로 요청할 때만).

```
product-planner → designer → senior-dev(Red) → 개발자(Green) → senior-dev(리뷰) → qa → devops
                                  ▲                    │              │
                                  └── 테스트 틀림 ──────┘   결함 되돌림 ─┘
```

## TDD 불변 규칙 (모든 위임에서 지킨다)

- **순서는 항상 테스트 → 코드.** 계약·공개 동작은 senior-dev가 테스트로 먼저 박고, 내부 구현 세부는 개발자가 직접 테스트 먼저(실패 확인) 후 구현.
- **계약 밖 변경 금지** — API 응답 필드를 임의로 추가/변경하면 다른 스택이 깨진다. 계약 변경은 senior-dev 통해서만.
- 백엔드 테스트 단위 = **컨트롤러 통합테스트만** (`code-convention` §6).

## 에이전트 간 데이터 전달 규칙

서브에이전트는 **격리된 컨텍스트**에서 돈다. 내가 준 것만 보고 요약만 돌려준다. 그래서:
- **넘길 때**: 다음 에이전트가 알아야 할 걸 명시적으로 다 적는다("아까 그거" 금지).
  - senior-dev 에게 → 기획서·인수 조건 원문
  - 개발자에게 → API 계약 + senior-dev 가 쓴 테스트 위치 + 관련 파일 경로
  - qa 에게 → 기획서 + 무엇이 구현됐는지(엔드포인트·화면)
- **받을 때**: 각 에이전트의 반환 형식대로 온다. 그대로 다음 단계 입력으로 쓴다.

## 실패·되돌림 처리

- 리뷰/qa 되돌림이 3번 넘게 반복되면 → 멈추고 사람에게 보고 (무한 루프 방지).
- 에이전트가 계약/설계를 벗어나거나 막히면 → 억지로 진행 말고 사람에게 결정을 넘긴다.
- 개발자가 "테스트 깨짐"을 보고하면 → 감추지 말고 그대로 사람에게 전달.
- **커밋·푸시·배포는 사용자가 명시적으로 요청할 때만** — 임의 실행 금지.

## 참고
- 코드 규칙 → `code-convention` 스킬 / 구조·모듈 편입 → `architecture` 스킬 / 배포 → `railway-deploy` 스킬.
- 순수 탐색·조사는 역할 에이전트 대신 **Explore/general-purpose** 서브에이전트가 빠르다.
